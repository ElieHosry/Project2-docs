CMS

Scope

Content authoring and publishing platforms — the editor/builder IS the product: authors create structured content, audiences consume the published output. The honest version runs on a C++/WASM live-preview renderer; strong teams only.

Examples

  1. Portfolio site builder — data-driven templates, real-time WYSIWYG preview renderer in C++/WASM, the builder itself is the full-stack tool. References:

    • Carrd
    • Carrd tutorial (YouTube)

  2. Restaurant menu & ordering CMS — owners edit menus/themes with live preview; published site served to customers; media processing service. References:

    • GloriaFood
    • GloriaFood tutorial (YouTube)

  3. News/blog platform with editorial workflow — draft → review → publish pipeline, C++ search/indexing service, article editor with live preview. References:

    • Ghost
    • Ghost staff roles / editorial workflow
    • Ghost tutorial (YouTube)

  4. Event microsite generator — organizers assemble event pages (schedule, speakers, registration) from data-driven blocks; live preview renderer; attendee counts via Firebase analytics. References:

    • Luma
    • Luma events tutorial (YouTube)

  5. Documentation / wiki knowledge base — markdown-first team wiki with live preview, document tree, collaborative editing, and fast published output. References:

    • Outline (official)
    • Outline demo tour (YouTube)
    • Outline (GitHub)

How each requirement applies

Common to all five: Custom Tools ✅ (the editor/builder IS the product, satisfying the "tools already full-stack" clause) · Data-Driven Architecture ✅ (templates and content models as data) · C++ Performant Service (preview/indexing/media engine) · Application Communication · Continuous Integration · Front-end + Local Backend · Authentication ✅ (author/owner accounts).

1. Portfolio builder

Platform fit: Web browser on mobile + Native PC (authoring) — editing on Native PC (or desktop web), published portfolios viewed via Web browser on mobile.
Front end: WYSIWYG editor UI, published portfolio site. Languages: HTML/JS/CSS + C++ preview renderer via WASM.
Back end: Layout/preview engine, content storage & templates, versioning, Firebase accounts. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): real-time WYSIWYG preview renderer in C++/WASM re-rendering on every edit.
  • 2D (adapted): custom layout/typography rendering.
  • External DB / Analytics ⏭️: sites, templates, visitor stats per portfolio.
  • Content versioning: undo history and named revisions of a site; visual diff between versions.

2. Restaurant menu & ordering CMS

Platform fit: Web browser on mobile + Native PC (authoring) — menu/theme editing on Native PC (or desktop web), customer browsing & ordering via Web browser on mobile.
Front end: Menu/theme editor, customer ordering site. Languages: HTML/JS/CSS + WASM preview.
Back end: Media processing service, order handling, menu versioning, Firebase. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): live theme/menu preview. C++ service: media processing (image resize, optimization).
  • 2D (adapted): theme preview rendering.
  • External DB / Analytics ⏭️: menus, orders, best-seller stats; auth for owner/customer roles.
  • Content versioning: menu revisions with rollback — "restore last week's menu" in one click.

3. News/blog platform

Platform fit: Web browser on mobile + Native PC (authoring) — article editing & editorial workflow on Native PC (or desktop web), readers via Web browser on mobile.
Front end: Article editor with live preview, reader site. Languages: HTML/JS/CSS + WASM preview.
Back end: Search/indexing service, editorial workflow engine, versioning, Firebase roles. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): C++ search/indexing service with incremental updates.
  • Custom tool: article editor with live preview + editorial workflow (draft → review → publish).
  • Networking: optional collaborative editing. External DB / Analytics ⏭️: articles, roles, readership stats.
  • Content versioning: the editorial backbone — draft history, reviewer diffs, publish rollback.

4. Event microsite generator

Platform fit: Web browser on mobile + Native PC (authoring) — page building on Native PC (or desktop web), published microsites viewed via Web browser on mobile.
Front end: Block-based event page editor, published microsites. Languages: HTML/JS/CSS + WASM preview.
Back end: Block preview renderer, registration handling, analytics, Firebase. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): block-based live preview renderer.
  • Data-driven: schedule/speakers/registration blocks as pure data.
  • External DB / Analytics ⏭️: registrations, attendance analytics.
  • Content versioning: staged site versions — edit next year's page while this year's stays live.

5. Documentation / wiki knowledge base

Platform fit: Web browser on mobile + Native PC (authoring) — editing on Native PC (or desktop web), reading via web browser on mobile.
Front end: Markdown editor, document tree, reader view. Languages: HTML/JS/CSS + C++ preview renderer via WASM.
Back end: Versioning/diff engine, collaboration sync, search index, Firebase roles. Languages: C++.
Backend scope: Server-indicated (T2).
  • Simulation loop (adapted): live markdown preview renderer (C++/WASM) re-rendering as you type.
  • Custom tools: the editor and document tree ARE the product. Data-driven: pages and site structure as data.
  • Import / export: markdown round-trip. Networking: collaborative editing (CRDT/OT basics).
  • External DB / Analytics ⏭️: documents, roles, readership stats.
  • Content versioning: page history and diffs — the most natural versioning showcase of all five examples.

Goals per Example

G1–G3 ⭐ are the common goals; the other 3 come from the CMS pool (# = pool number).

1. Portfolio site builder

M1:

  • G1 ⭐ Application Shell, Real-time C++ Simulation Loop, Input & Core Logic
    Tech & dependencies
    • Tech: C++ layout core compiled to WASM (Emscripten), edit-loop re-rendering on change; CMake, Git.
    • Prerequisites: None — foundation of this project.
    • Feeds into: M1 #4 (preview engine is its main consumer); every other goal.
  • G2 ⭐ Data-Driven Architecture of Core Objects / Entity System
    Tech & dependencies
    • Tech: Sites, pages, sections and blocks modeled as JSON entities (nlohmann/json) with hot-reload, so a portfolio is pure data interpreted by the renderer.
    • Prerequisites: G1 (shell & loop).
    • Feeds into: M1 #4 and #12 render this model; M2 G2 editor and M2 #7 versioning operate on it.
  • G3 ⭐ Debugging, Profiling, Stress Test & CI
    Tech & dependencies
    • Tech: Tracy profiling of re-render times, Catch2 layout unit tests, stress tests with large multi-section portfolios, GitHub Actions Windows + Linux matrix.
    • Prerequisites: G1 (loop to instrument).
    • Feeds into: M2 G3 (CI advanced); keeps M1 #4/#5 render performance honest.
  • #4 Live Preview Rendering Engine
    Tech & dependencies
    • Tech: C++ layout engine in WASM with dirty-region re-render so the portfolio page updates on every edit, FreeType text metrics.
    • Prerequisites: G1 (edit loop) + G2 (page/block model).
    • Feeds into: M2 G2 editor binds to it; M2 #12 runtime reuses the same renderer for published sites.
  • #5 2D Layout & Typography Rendering
    Tech & dependencies
    • Tech: Custom text layout and typography via FreeType plus stb_image for photo/gallery rendering — the visual-quality layer of portfolio pages.
    • Prerequisites: G1 (shell & loop).
    • Feeds into: M1 #4 draws through it; M2 #12 published pages inherit its fidelity.
  • #12 Theming System
    Tech & dependencies
    • Tech: Themes (palettes, font sets, spacing scales) as hot-reloadable, schema-validated JSON applied at render time.
    • Prerequisites: G2 (data-driven architecture).
    • Feeds into: M1 #4/#5 restyle instantly on theme switch; M2 #12 ships the theme with the published site.

M2:

  • G1 ⭐ Architecture — Optimization & Platform Validation
    Tech & dependencies
    • Tech: Tracy profiling of the WASM preview, ASan on MSVC, memory pools for layout nodes; Emscripten builds validated in desktop and mobile browsers.
    • Prerequisites: All M1 goals (hardens the M1 renderer and data model).
    • Feeds into: M2 #12 runtime performance; M2 #6 editor responsiveness.
  • G2 ⭐ Content Editor — Core
    Tech & dependencies
    • Tech: WYSIWYG site-editor core — web front-end (HTML/JS/CSS) over the WASM renderer, JSON serialization, undo/redo command stack.
    • Libraries/Software: HTML/JS/CSS (editor UI), Emscripten (WASM live preview of the C++ renderer), nlohmann/json (C++-side serialization)
    • Prerequisites: M1 G2 (data model) + M1 #4 (live preview to bind to).
    • Feeds into: M2 #6 advanced tool features; M2 #7 hooks revisions into its edit stack.
  • G3 ⭐ CI — CMake, Code Quality, Build Standards & Packaging
    Tech & dependencies
    • Tech: CMake presets, clang-format + clang-tidy, packaged builder plus CI deploy of generated portfolio sites to Firebase Hosting.
    • Prerequisites: M1 G3 (CI foundation).
    • Feeds into: Release builds of every goal; M2 #12 deploy path.
  • #6 Editor — Advanced Tool Features
    Tech & dependencies
    • Tech: Drag-drop section reordering, inline text editing and alignment/snapping guides bound to the live WASM preview.
    • Prerequisites: M2 G2 (editor core) + M1 #4 (preview binding).
    • Feeds into: Content-creation showcase; exercises M2 #7 undo/rollback history.
  • #7 Content Versioning
    Tech & dependencies
    • Tech: Site snapshots with Myers diff and JSON revision metadata — named revisions of a portfolio plus visual diff between versions.
    • Prerequisites: M1 G2 (site/page data model to snapshot).
    • Feeds into: M2 G2/#6 undo history and rollback; safe re-publishing via M2 #12.
  • #12 Published-Site Runtime
    Tech & dependencies
    • Tech: Static generation reusing the WASM layout renderer, sitemap/meta generation, Firebase Hosting — fast portfolio pages on mobile browsers.
    • Prerequisites: M1 #4 (renderer) + M1 #12 (themes) + M2 G1 (optimization).
    • Feeds into: The visitor-facing deliverable; endpoint of M2 #7 staged publishing.
