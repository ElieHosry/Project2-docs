Education + Training

Scope

Products that teach or train — interactive lessons, skill drills, lab simulations and trainers, with learners (and usually teachers/authors) as first-class roles. The engineering must outweigh the content authoring: the interactive engine is the deliverable, not the lesson library.

Examples

  1. Typing / coding trainer — real-time keystroke analysis, adaptive difficulty engine in C++, lesson editor; progress stored in Firebase. References:

    • Monkeytype
    • keybr — adaptive lessons

  2. Interactive circuit-building lab — students wire virtual electronic circuits; C++ loop simulates current flow in real time; component/exercise editor. References:

    • PhET Circuit Construction Kit: DC
    • Runnable HTML5 virtual lab

  3. Language-learning mini-game platform — spaced-repetition engine, gamified drills, content editor for teachers to author word packs. References:

    • Duolingo
    • Duolingo teaching method

  4. Driving theory trainer — interactive traffic scenarios simulated in C++; students react to hazards in real time; scenario editor for instructors; scores in Firebase. References:

    • DVSA hazard perception practice (GOV.UK)
    • Official DVSA hazard perception guide (YouTube)

  5. Music ear-training app — interval/chord/rhythm recognition drills with real-time audio playback and animated feedback; exercise editor for teachers. References:

    • Tenuto by musictheory.net
    • EarMaster (official)
    • EarMaster intro (YouTube)

How each requirement applies

Common to all five: C++ Performant Service (the learning/simulation engine) · Application Communication (UI ↔︎ engine) · Continuous Integration (CMake, tests, packaging) · Front-end + Local Backend · Authentication ✅ (learner accounts and progress are central to the category).

1. Typing / coding trainer

Platform fit: Web browser on mobile — WASM build; a Bluetooth keyboard pairs with the phone for serious typing drills (on-screen keyboard for casual practice). The PC build is the development vehicle.
Front end: Text & feedback rendering (FreeType), per-key heat map UI. Languages: C++, GLSL.
Back end: Keystroke capture engine, adaptive difficulty model, Firebase progress sync. Languages: C++.
Backend scope: Local only.
  • Simulation loop (adapted): real-time keystroke capture, latency measurement, adaptive difficulty engine.
  • Data-driven: lessons, drills, and difficulty curves defined in data. Custom tool: lesson editor.
  • External DB / Analytics ⏭️: per-user progress, streaks, class-level analytics.
  • 2D: custom text/feedback rendering (per-key accuracy heat colors).
  • Localization pipeline: keyboard layouts (QWERTY/AZERTY) and language lesson packs swappable as content.

2. Interactive circuit-building lab

Platform fit: Web browser on mobile + Native mobile — touch drag-and-drop wiring on the phone; the PC build is used for development.
Front end: Schematic canvas, drag-and-drop component UI; web build shell. Languages: C++, GLSL; HTML/JS/CSS (web shell via WASM).
Back end: Current-flow solver, exercise engine, saved circuits. Languages: C++.
Backend scope: Local only.
  • Simulation loop: continuous current-flow simulation with live component state — closest to a true physics loop.
  • Data-driven: components and exercises as data assets. Custom tool: component/exercise editor.
  • External DB / Analytics ⏭️: saved circuits, exercise completion stats for teachers.
  • 2D: animated schematic rendering (wires, current flow, meters).
  • Localization pipeline: exercise text and component labels translated as data, no code changes.

3. Language-learning mini-game platform

Platform fit: Native mobile + Web browser on mobile — short daily sessions on the phone.
Front end: Mini-game screens, feedback animation, lesson navigation. Languages: C++, GLSL; HTML/JS/CSS (web shell via WASM).
Back end: Spaced-repetition scheduler, drill logic, content engine, Firebase sync. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): spaced-repetition scheduler + real-time mini-game logic in C++.
  • Data-driven: word packs, drills, and game rules authored as content. Custom tool: teacher content editor.
  • External DB / Analytics ⏭️: progress sync, streaks, teacher dashboards.
  • 2D: mini-game rendering and feedback animation.
  • Localization pipeline: inherently core — course content, UI, and audio all flow through it.

4. Driving theory trainer

Platform fit: Native mobile + Web browser on mobile — tap-to-react hazard drills suit touch.
Front end: Traffic scene rendering, tap/reaction capture UI. Languages: C++, GLSL; HTML/JS/CSS (web shell via WASM).
Back end: Traffic scenario simulation, reaction scoring, Firebase results. Languages: C++.
Backend scope: Local only.
  • Simulation loop: traffic scenarios (vehicles, pedestrians, hazards) simulated in real time; reaction timing scored against it.
  • Data-driven: scenarios as data. Custom tool: instructor scenario editor.
  • External DB / Analytics ⏭️: scores, per-scenario failure rates for instructors.
  • 2D: road/traffic scene rendering. 3D: optional pseudo-3D road view.
  • Localization pipeline: country-specific traffic rules and signage as swappable content packs.

5. Music ear-training app

Platform fit: Native mobile + Web browser on mobile — short practice sessions anywhere, headphones on.
Front end: Staff/keyboard rendering, answer UI, feedback animation. Languages: C++, GLSL; HTML/JS/CSS (web shell via WASM).
Back end: Audio synthesis/playback engine, assessment engine, progress sync. Languages: C++.
Backend scope: Local only.
  • Simulation loop: real-time audio synthesis/playback with precise timing; answer scoring against it.
  • Data-driven: exercises and progressions as data. Custom tool: teacher exercise editor.
  • External DB / Analytics ⏭️: progress, per-interval accuracy stats for teachers. Auth: learner accounts.
  • 2D: staff/keyboard rendering, animated feedback.
  • Localization pipeline: solfège vs. letter naming, translated exercise content as packs.

1. Typing / coding trainer

M1:

  • G1 ⭐ Application Shell, Real-time C++ Simulation Loop, Input & Core Logic
    Tech & dependencies
    • Tech: SDL2 window and raw keyboard scancode capture, fixed-timestep loop timestamping every keystroke at millisecond precision; C++17/20, CMake, Git on Windows (MSVC).
    • Prerequisites: None — foundation of this project.
    • Feeds into: M1 #5 (interaction engine consumes the timed input stream); every other goal.
  • G2 ⭐ Data-Driven Architecture of Core Objects / Entity System
    Tech & dependencies
    • Tech: Lessons, drills, word lists, and difficulty curves defined as hot-reloadable JSON so lesson content changes without recompiling; C++ serialization and reflection patterns.
    • Prerequisites: G1 (shell and loop to load content into).
    • Feeds into: M1 #7 (difficulty model reads lesson data), M1 #10 (progress schema), M2 G2 (lesson editor edits this format).
  • G3 ⭐ Debugging, Profiling, Stress Test & CI
    Tech & dependencies
    • Tech: ImGui overlay showing live WPM, latency, and error stats; Tracy profiling of input-to-feedback latency; Catch2 tests replaying synthetic keystroke streams; GitHub Actions Windows and Linux builds.
    • Prerequisites: G1 (running loop to instrument).
    • Feeds into: M2 G3 (CI and packaging); validates M1 #5 timing accuracy under stress.
  • #5 Real-time Interaction Engine
    Tech & dependencies
    • Tech: Keystroke event pipeline mapping SDL2 scancodes to expected characters, per-key latency measurement and error classification, streak and combo detection in C++.
    • Prerequisites: G1 (raw input and loop timing).
    • Feeds into: M1 #7 (per-key stats drive the difficulty model) and M1 #10 (results persisted to Firebase).
  • #7 Adaptive Difficulty Engine
    Tech & dependencies
    • Tech: Per-key confidence model (keybr-style) with decay curves selecting which characters and words the next drill emphasizes, custom C++.
    • Prerequisites: G2 (lesson data) and M1 #5 (accuracy and latency measurements).
    • Feeds into: M2 #8 (Adaptive Engine II extends the model) and M2 #10 (its decisions get analyzed).
  • #10 Progress Tracking & Firebase
    Tech & dependencies
    • Tech: Firebase Auth and Firestore via REST from C++ storing per-user WPM history, per-key accuracy, and streaks, with offline queueing and sync.
    • Prerequisites: G2 (progress data model) and M1 #5 (results to store).
    • Feeds into: M2 #10 (analytics aggregates this data) and M2 #8 (learner model reads the history).

M2:

  • G1 ⭐ Architecture — Optimization & Platform Validation
    Tech & dependencies

  • G3 ⭐ CI — CMake, Code Quality, Build Standards & Packaging
    Tech & dependencies
    • Tech: CMake presets, clang-format and clang-tidy, GitHub Actions matrix builds, CPack packaging of the Windows trainer plus the Emscripten web bundle.
    • Prerequisites: M1 G3 (CI foundation).
    • Feeds into: Showcase releases of the trainer and the M2 G2 lesson editor.
  • #8 Adaptive Engine II
    Tech & dependencies
    • Tech: Full learner model with per-key, per-bigram, and per-word confidence plus difficulty curves, evaluated against logged practice sessions.
    • Prerequisites: M1 #7 (adaptive engine) and M1 #10 (logged session history).
    • Feeds into: M2 #10 (insights compare model predictions to real outcomes) and the personalized showcase demo.
  • #9 Localization & Content Pipeline
