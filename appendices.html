A. Backend Scope Levels

Every example's Back end box states one of three backend scopes. Note that Firebase is remote backend-as-a-service and does not count as a student-built server; the levels below refer to a custom server backend — the team's own C++ service running on the Linux server platform (planned for the second trimester).

Local only — the project works completely and correctly with only a local backend. Nothing about the product demands a custom server; a remote server would add little beyond what Firebase already provides. Example: the typing trainer or the museum installation.

Local now, server-ready (T2) — the project works fully in trimester 1 with a local backend, but the architecture has a clean seam where the team could lift its C++ service onto the Linux server in trimester 2 for full-stack practice. Moving it is a choice made for learning value, not because the product breaks without it. Example: the traffic simulator — the simulation engine could run server-side with the browser as a thin viewer.

Server-indicated (T2) — the product's core promise inherently involves multiple users sharing live state, and a local backend is only a stopgap. In trimester 1 the team can bridge with Firebase, but the real product genuinely requires an authoritative server process. Teams choosing these projects are implicitly committing to server work in trimester 2. Example: the pixel canvas — hundreds of users painting one shared canvas with rate limiting needs a central server.

Put simply: server-ready = "the door is open if you want it"; server-indicated = "the design is knocking on that door."

B. Colored Box Legend

  • Blue — Platform fit: the recommended target platform(s) for the example (see Appendix F for the three platform options).
  • Green — Front end: the front-end components and their programming languages.
  • Amber — Back end: the back-end components, their programming languages, and the backend scope (see Appendix A).
  • Purple text: the newly suggested core tech requirement for the category (last row of each matrix).

C. Goal Sizing & Schedule Guidance

Sizing. A goal is a heavy component sized at ~40 hours per student per milestone. With Milestone 1 at week 9 and Milestone 2 at week 13, both phases carry a uniform ~10 hours/week (of which ~4 are lab-hours). Within a pool, goals are not perfectly equal in weight — at approval time, instructors should push back on teams cherry-picking three light customizable goals.

Foundation sprint (weeks 4–5). After the week-4 proposal, teams start project-agnostic work immediately, before formal approval: the G1 owner builds the application skeleton (window, fixed-timestep loop, input, CMake), the G3 owner sets up the repository, CI, and test harness, the G2 owner drafts provisional data schemas, and the remaining members run small spikes on their candidate goals (render a sprite, play a sound, open a socket). Almost none of this work is wasted if the proposal is modified at approval. Week-5 approval remains a real gate: the project direction is still pending until confirmed.

G1 skeleton first. Nearly every goal depends on the application shell and loop. The G1 owner must deliver an interface-first skeleton within the first week of the foundation sprint so the other five members never start against a blank repository.

Integration is part of every goal. Six parallel components produce six islands unless integration is explicit. "Integrated into the main build and demoed inside the team's application" is part of every goal's definition of done; budget roughly 20% of each goal's hours for it. The weekly follow-up around week 7 doubles as an integration checkpoint (ungraded — the build runs and everyone's component is in it), preventing integration debt from hiding until Milestone 1.

Team due diligence. Instructors follow up with each team on a weekly basis: progress against each member's goal, blockers, and integration health. Weekly visibility surfaces pacing and bus-factor problems within days instead of at milestone demos, and gives the foundation sprint and definition-of-done rules teeth without extra graded gates.

Bus factor. One goal per owner means one struggling student can silently remove a subsystem. Use cross-review pairs: each student reviews one neighbor's goal, keeping at least two people minimally fluent in every component.

D. Category Suitability Ranking

Ranking of the 9 categories from most to least suitable for this module, judged against the core requirements (real-time C++ simulation loop, 2D graphics, custom tools, low-level libraries, data-driven architecture) and against second-year team realities (skill level, motivation, demo-ability). This is an instructor-facing assessment, not a rule — a strong team can succeed in any category.

1. Games — every core requirement applies directly with zero reinterpretation; the milestone example was written for it; motivation and reference material are unmatched. The risk is scope creep, which the fixed goal structure already contains.

2. Immersive Experience — technically a near-twin of Games (loop, 2D, audio, scene editors all direct) with more artistic freedom and less genre pressure. Slightly narrower product scope, and installation-style projects may depend on specific hardware for the final demo.

3. Creative Tools & Media Production — the tool IS the product, satisfying the tools-are-full-stack clause with zero reinterpretation; real-time canvas/audio/particle loops and heavy custom 2D rendering rival Games for requirement fit, and the non-destructive edit history teaches serious architecture. Risks: professional tools have bottomless feature scope (contained by the fixed goal structure), and UX polish can compete with engineering time.

4. Simulator — the simulation loop is not just satisfied but is the deliverable; headless batch mode as its core (purple) requirement adds scientific rigor to every project. Visuals are less flashy, so team motivation varies; networking is rarely exercised.

5. Fintech — a sleeper fit: high-frequency feeds map cleanly onto the loop in the feed-driven examples (✅ in 2 of 5, 🔶 adapted elsewhere), custom chart rendering forces real 2D work, and the ledger teaches engineering rigor. Costs: 2D and tools need domain adaptation (🔶), and the domain can feel dry to some students.

6. Education + Training — solid fit with natural auth/analytics use, and two examples (circuit lab, driving theory) are true simulations. But the loop is adapted in the others, and content authoring competes with engineering time — teams can sink hours into lesson content that teaches them little engineering.

7. Social Media — networking practice is excellent and unique to this category, but the high-frequency C++ loop is a stretch everywhere except the pixel canvas; Firebase can quietly swallow the backend, undermining the C++ requirement; and the products are hard to demo convincingly without a real user base. Moderation/safety adds overhead that isn't core engineering.

8. GovTech — the simulation-heavy examples (queue management, transport tracker) fit well, but the categories' flagship-sounding workflows (permits, forms) satisfy the loop only weakly, and the graphical output is mostly dashboards and forms. Domain access and student motivation are practical limits — though the category's scope explicitly includes private institutional/enterprise services (universities, hospitals, companies), which puts familiar domains like campus workflows within easy reach.

9. CMS — the weakest alignment with the raw requirements: the simulation loop must be reinterpreted as a preview renderer (or indexing service) in every example, graphics means layout/typography rather than real-time rendering, and the honest version of that (a C++/WASM live preview engine) is genuinely hard — likely beyond an average second-year team — while the dishonest version collapses into ordinary web development, which this module is explicitly not about.

Suggestions

  • CMS — remove or restrict. Either drop it, or keep it as a strong-teams-only category (the C++/WASM preview renderer is the only version that meets the module's intent, and it is advanced). If dropped, its best ideas survive: the wiki's versioning/collaboration lives on in other categories' goals.
  • GovTech — keep, but steer. Guide teams toward the simulation-heavy examples (queue management, transport tracker, city dashboard) and treat the permit workflow as approval-with-caution, since its loop fit is the weakest in the catalog.
  • Social Media — keep with a guard-rail. Require the C++ service to be demonstrably load-bearing (e.g., the pixel canvas state engine or a real ranking engine benchmarked in CI), so Firebase does not become the de-facto backend.
  • Creative Tools & Media Production — adopted as the 9th category (originally proposed here as a CMS replacement; added without removing CMS). Paint/animation tools, audio editors, music trackers, VFX editors. It fits the module almost as well as Games — and remains the natural home for the "strong CMS team" profile without the web-dev gravity. See its category page and pool.
  • Second candidate: Robotics & Embedded Simulation — virtual robot arenas, line-followers, drone/rover simulators. Physics-loop-first like Simulator, with strong industry appeal; needs no hardware since everything is simulated.

E. Windows Development Environment Notes

Development happens on Windows machines using Visual Studio or VS Code. The module's stack is deliberately cross-platform — SDL2, OpenGL, FMOD/OpenAL, FreeType, ImGui, CMake, and Git are all first-class on Windows, and CMake generates Visual Studio solutions natively — so no goals or categories change. The following notes prevent Windows-specific surprises:

  • CI builds on Windows and Linux from day one. The trimester-2 server targets Linux; a dual-OS GitHub Actions matrix (windows-latest + ubuntu-latest) catches MSVC-only code while it is still cheap to fix. "Builds green on both platforms" is part of the G3/CI goal's definition of done.
  • Sanitizers: MSVC supports AddressSanitizer only; UndefinedBehaviorSanitizer requires clang-cl or WSL. Goal entries mentioning sanitizers assume ASan on MSVC as the default.
  • Sockets: on Windows, raw sockets mean Winsock2, not BSD sockets. Teams using ENet or SDL_net get this abstracted away — recommended for networking goals.
  • Emscripten/WASM builds: fully supported on Windows via emsdk, but pair better with VS Code + CMake/Ninja than with the Visual Studio IDE. Teams targeting web builds should adopt that workflow early.
  • Android: the NDK, emulator, and device deployment all work normally on Windows hosts.
  • WSL (optional): a convenient escape hatch for Linux-only tooling (UBSan, server smoke tests) without leaving the Windows machine; not required in trimester 1.

F. Platform Fit Options

Every example's blue Platform fit box names the platform(s) the finished product ships on. Policy: Native PC is the development, debugging and first-testing vehicle for every team — it is not a shipping target on its own. Every project's showcase deliverable must run on at least one mobile output: Native mobile (reference device: Google Pixel 8a / 9a) and/or Web browser on mobile. Native PC may appear in a platform fit only as the operator/authoring side of a role split (marked "(ops)" or "(authoring)"). All targets share the same C++ codebase — the platform choice affects the build toolchain, input model, and demo story, not the architecture.

1. Native PC (Windows) — the development vehicle. Visual Studio / VS Code, CMake, the fastest build-run-debug loop, and full tooling support (ImGui overlays, Tracy, ASan — see Appendix E). Every team lives here day-to-day regardless of target. As a shipping platform it appears only in role splits: the staff dashboard, control-room display, or content-authoring desk of a product whose public side ships on mobile.

2. Native mobile (Android — reference device Google Pixel 8a / 9a). Built with the Android NDK; the emulator serves during development and the reference phone is the measurement target — frame budgets, memory ceilings and audio latency are validated on it, and acceptance criteria may cite it by name ("60 FPS on the reference device"). Implies touch-first input design and APK packaging in CI. AAudio gives a credible low-latency path for audio-centric products. On-device validation belongs to M2 G1 (Platform Validation).

3. Web browser on mobile. The C++ core compiled to WASM via Emscripten, wrapped in an HTML/JS/CSS shell. Distribution is a URL — nothing to install, ideal for reaching many users instantly (shared canvases, citizen-facing services, published content, dashboards). Trade-offs: lower performance headroom than native, browser sandbox restrictions (WebSockets instead of raw sockets), and a VS Code + CMake/Ninja workflow on Windows (Appendix E).

Combinations. Two patterns occur: same app on multiple mobile targets (e.g. "Native mobile + Web browser on mobile" — the cross-platform stack makes this cheap, and M2 G1 is where the secondary target is proven), and role splits (e.g. "Web browser on mobile + Native PC (ops)" — the public side ships on mobile, the operator/authoring side runs on PC). When in doubt, teams should pick the mobile target by input model and demo story — and remember the PC build remains their iteration platform throughout, shipping or not.

Enforcement hooks. The mobile/WASM build job belongs in the G3 CI matrix from week one, so the mobile target never becomes a late surprise; "the showcase runs on the mobile target" is part of M2 G1's definition of done.
