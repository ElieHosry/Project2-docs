Simulator

Scope

Products whose deliverable IS the simulation — modeling a real-world system (traffic, ecosystems, crowds, epidemics, economics) to observe, analyze and experiment. Scientific rigor is the signature: seeded reproducibility, statistics, batch experiments.

Examples

  1. Traffic intersection simulator — vehicles, lights, and pedestrians simulated in C++; scenario editor to design road layouts; statistics dashboard. References:

    • Intersection Controller (Google Play)
    • Mini Motorways — trailer (YouTube)

  2. Ecosystem / predator-prey simulator — agent-based simulation with tunable parameters, real-time 2D visualization, experiment configuration tool. References:

    • NetLogo Wolf Sheep Predation
    • Sebastian Lague — Simulating an Ecosystem (YouTube)

  3. Elevator system simulator — multi-elevator scheduling in a busy building; algorithm comparison mode; building/traffic-pattern editor. References:

    • Elevator Saga (playable)
    • Elevator simulator visualization in C++/SDL (YouTube)

  4. Crowd evacuation simulator — agent-based crowd flow through a floor plan (panic, bottlenecks, exits); floor-plan editor; run statistics to Firebase. References:

    • Pathfinder by Thunderhead
    • Pathfinder demos (YouTube playlist)

  5. Epidemic spread simulator — agent-based SIR infection model with tunable scenarios (distancing, quarantine, travel); alert thresholds; batch policy experiments. References:

    • Washington Post — "simulitis" corona simulator
    • 3Blue1Brown — Simulating an epidemic (YouTube)
    • NetLogo epiDEM Travel and Control

How each requirement applies

Common to all five: the Simulation Loop is the core deliverable (true real-time agent simulation) · C++ Performant Service (the simulation engine) · Application Communication (UI ↔︎ engine) · Continuous Integration · Front-end + Local Backend · Data-Driven Architecture (scenarios and parameters as data) · Auth 🔵 optional (saved experiments per user).

1. Traffic intersection

Platform fit: Web browser on mobile + Native PC (authoring) — running/viewing on the phone; scenario authoring on the PC side.
Front end: Top-down world view, run controls (ImGui), live charts. Languages: C++, GLSL.
Back end: Agent simulation core (vehicles, lights, pedestrians), statistics engine, Firebase run storage. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop: vehicle/pedestrian agents, traffic-light logic. 2D: top-down road rendering.
  • Custom tool: road-layout scenario editor. External DB / Analytics ⏭️: run statistics, scenario sharing.
  • 3D: optional isometric view.
  • Headless batch mode: sweep light-timing parameters over hundreds of runs; report throughput/wait-time stats.

2. Ecosystem / predator-prey

Platform fit: Web browser on mobile — WASM build to configure and watch runs on the phone; large batch experiments run on the PC development build.
Front end: World/population rendering, parameter sliders, live charts. Languages: C++, GLSL.
Back end: Agent behavior simulation, reproducible seeded runs, experiment results storage. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop: agent behaviors (hunger, reproduction, sensing) with tunable parameters. 2D: world/population rendering with live charts.
  • Custom tool: experiment configuration + parameter editor. External DB / Analytics ⏭️: experiment results for comparison.
  • Headless batch mode: the natural fit — population-stability studies across seeds and parameters.

3. Elevator system

Platform fit: Web browser on mobile — lightweight visualization is web-first; the PC build is used for development.
Front end: Shaft/floor/queue visualization, run controls; web viewer shell. Languages: C++, GLSL; HTML/JS/CSS (viewer via WASM).
Back end: Elevator scheduling simulation, discrete-event queue, algorithm benchmarks. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop: multi-elevator scheduling under generated passenger traffic; algorithm A/B comparison mode.
  • Custom tool: building/traffic-pattern editor. 2D: shaft/floor/queue visualization.
  • External DB / Analytics ⏭️: algorithm benchmark results.
  • Headless batch mode: compare scheduling algorithms statistically (avg/max wait) without animation.

4. Crowd evacuation

Platform fit: Web browser on mobile + Native PC (authoring) — watch evacuations on the phone; floor-plan authoring on the PC side.
Front end: Floor-plan view, density/heatmap overlays, editor UI (ImGui). Languages: C++, GLSL.
Back end: Crowd-flow agent simulation, panic models, run statistics. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop: agent flow, bottlenecks, exit choice under panic parameters. 2D: floor-plan + agent/density rendering.
  • Custom tool: floor-plan editor. External DB / Analytics ⏭️: run statistics. 3D: optional isometric view.
  • Headless batch mode: evacuation-time distributions across layouts — the safety-analysis use case.

5. Epidemic spread simulator

Platform fit: Web browser on mobile — WASM build for scenario setup and viewing; heavy batch sweeps run on the PC development build.
Front end: Dot-agent world, live SIR curves, parameter controls (ImGui). Languages: C++, GLSL.
Back end: SIR infection simulation, alert thresholds, headless batch runner, Firebase results. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop: agent movement, transmission probability, SIR state transitions per tick.
  • Data-driven: scenarios (infection rates, policies) as data. Custom tool: scenario/parameter editor.
  • 2D: dot-agent world plus live SIR curves. Alerts: threshold warnings (e.g., hospital capacity).
  • External DB / Analytics ⏭️: run results and scenario sharing.
  • Headless batch mode: policy comparisons across hundreds of seeded runs — the showcase result.

    • Prerequisites: G1 (shell & loop).
    • Feeds into: M1 G15 (road layouts as data), M1 G5 (agent definitions), M2 G2/G6 (editor edits this data).
  • G3 ⭐ Debugging, Profiling, Stress Test & CI
    Tech & dependencies
    • Tech: ImGui overlays for queue lengths and light phases, Tracy profiling of rush-hour agent counts, Catch2 tests, GitHub Actions Windows + Linux builds.
    • Prerequisites: G1 (shell & loop).
    • Feeds into: M2 G3 (CI advanced); M2 G7 (stress-test hooks reused for batch runs).
  • G5 Agent Behavior System
    Tech & dependencies
    • Tech: Finite-state machines for vehicles (approach/queue/turn) and pedestrians, traffic-light controller logic, lane-following steering in custom C++.
    • Prerequisites: G1 (tick) and G2 (agent definitions).
    • Feeds into: M1 G8 (produces throughput/wait metrics); M2 G7 (behavior exercised in batch runs).
  • G8 Statistics & Live Charts
    Tech & dependencies
    • Tech: Custom line charts of intersection throughput and average wait time rendered with SDL2/OpenGL and FreeType labels, incremental redraw per tick.
    • Prerequisites: G1 (loop); M1 G5 supplying the metrics.
    • Feeds into: M2 G4 (advanced visual layers); M2 G7 (same statistics power batch reports).
  • G15 Map / Environment System
    Tech & dependencies
    • Tech: Road network (lanes, intersections, crosswalks) as tiled JSON data with pan/zoom camera and spatial indexing for agent lookups.
    • Prerequisites: G2 (data model) plus the top-down rendering in G1's shell.
    • Feeds into: M1 G5 (agents navigate it); M2 G2/G6 (editor authors these layouts).

M2:

  • G1 ⭐ Architecture — Optimization & Platform Validation
    Tech & dependencies

  • G3 ⭐ CI — CMake, Code Quality, Build Standards & Packaging
    Tech & dependencies
    • Tech: CMake presets, clang-format + clang-tidy, packaging of the native app and Emscripten web viewer via GitHub Actions matrix builds.
    • Prerequisites: M1 G3 (CI foundation).
    • Feeds into: Release/showcase builds of every goal.
  • G4 Visualization — Advanced
    Tech & dependencies
    • Tech: GLSL congestion heat/density overlays on the road network, time scrubbing of a run, optional isometric view.
    • Prerequisites: M1 G15 (world rendering) and M1 G8 (metric data); M2 G1 (optimization).
    • Feeds into: Showcase visuals; live preview inside M2 G6.
  • G6 Scenario Editor — Advanced
    Tech & dependencies
    • Tech: Drag-drop road pieces, live simulation preview bound to the editor, deep undo/redo, and layout validation in ImGui.
    • Prerequisites: M2 G2 (editor core).
    • Feeds into: Content-creation showcase; scenario files fed to M2 G7 sweeps.
  • G7 Headless Batch Mode & Reports
    Tech & dependencies
    • Tech: CLI-driven std::thread run farm sweeping light-timing parameters over hundreds of seeded runs, CSV/JSON throughput and wait-time reports, results stored to Firebase.
