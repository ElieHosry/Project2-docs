Games

Scope

Interactive entertainment — any product whose core is a real-time game loop: action, puzzle, strategy, tower defense, rhythm and beyond, single- or multiplayer. If the fun comes from systems running at frame rate, it belongs here.

Examples

  1. Top-down dungeon crawler — procedurally generated levels, entity system, physics & collision, level editor as the custom tool. References:

    • The Binding of Isaac: Rebirth (Steam)
    • Enter the Gungeon — trailer (YouTube)

  2. 2D physics puzzle game (Angry Birds / Cut the Rope style) — physics simulation is the core loop; puzzle editor for content creation. References:

    • Cut the Rope (official)
    • Angry Birds Classic — gameplay (YouTube)

  3. Real-time multiplayer arena game — small networked matches (2–4 players), authoritative C++ server logic, matchmaking via Firebase. References:

    • Astro Party (App Store)
    • Astro Party — trailer (YouTube)

  4. Tower defense game — wave simulation, pathfinding, and enemy AI in the C++ loop; map/wave editor as the custom tool; online leaderboards. References:

    • Kingdom Rush (official)
    • Kingdom Rush — gameplay (YouTube)

  5. Rhythm action game — beat-synced note timing, deterministic input capture, animation-heavy feedback; beatmap editor as the custom tool. References:

    • osu! (official)
    • osu! beatmap editor (wiki)
    • Muse Dash — gameplay (YouTube)

How each requirement applies

Common to all five: C++ Performant Service (the game engine itself) · Application Communication (UI ↔︎ C++ engine messaging) · Continuous Integration (CMake, tests, build packaging) · Front-end + Local Backend (game front-end over the local C++ engine).

1. Top-down dungeon crawler

Platform fit: Native mobile — twin-stick touch controls on the reference phone (Pixel 8a/9a); the Native PC build is the development/iteration vehicle.
Front end: HUD & menus, sprite/tile renderer (SDL2/OpenGL), input & camera. Languages: C++, GLSL.
Back end: Engine core (simulation loop, entity system, physics, AI, procedural generation), save/persistence, Firebase sync. Languages: C++.
Backend scope: Local only.
  • Simulation loop: game loop driving entity updates, physics & collision, enemy AI.
  • Data-driven: rooms, items, enemies, and generation rules defined in data files.
  • Custom tool: room/level editor feeding the procedural generator.
  • External DB: cloud saves, leaderboards. Auth ⏭️: optional accounts. Analytics ⏭️: run statistics, death heatmaps.
  • 2D: sprite/tile renderer with lighting. 3D: optional 2.5D. Networking: not needed (single-player).
  • Replay System: deterministic loop enables run recording/playback — great for bug repro and share-a-run.

2. 2D physics puzzle game

Platform fit: Native mobile + Web browser on mobile — touch-first slingshot/drag interactions.
Front end: Touch UI, level renderer, particles; web build shell. Languages: C++, GLSL; HTML/JS/CSS (web shell via WASM).
Back end: Physics engine, level logic, progress & level-sharing services. Languages: C++.
Backend scope: Local only.
  • Simulation loop: the physics engine (rigid bodies, constraints, collisions) IS the core loop.
  • Data-driven: puzzle levels fully described in data; no hard-coded levels.
  • Custom tool: puzzle editor with live physics preview.
  • External DB: level sharing, progress sync. Auth ⏭️: optional. Analytics ⏭️: per-level completion/star rates.
  • 2D: physics-object rendering, particles. Networking / 3D: not needed — effort goes into physics quality.
  • Replay System: solution replays per level — validates determinism of the physics engine.

3. Real-time multiplayer arena game

Platform fit: Native mobile — single-device party play and networked matches on phones; the PC build is used for development and network testing.
Front end: Arena renderer, lobby & match UI, prediction display. Languages: C++, GLSL.
Back end: Authoritative simulation server, netcode (sockets/ENet), matchmaking via Firebase. Languages: C++.
Backend scope: Server-indicated (T2).
  • Simulation loop: authoritative C++ game state simulation at fixed tick rate.
  • Networking: core requirement — client prediction, state sync, 2–4 player matches.
  • External DB: matchmaking, match history. Auth: applies directly (player accounts). Analytics ⏭️: match/balance stats.
  • Custom tool: arena/map editor. Data-driven: ships/weapons/arenas as data assets.
  • 2D: fast arena renderer. 3D: optional.
  • Replay System: match replays from the network input stream — doubles as a desync debugging tool.

4. Tower defense game

Platform fit: Native mobile + Web browser on mobile — tap-to-place is ideal on touch.
Front end: Map/tower placement UI, HUD, wave feedback; web build shell. Languages: C++, GLSL; HTML/JS/CSS (web shell via WASM).
Back end: Wave simulation, pathfinding, balancing data services, leaderboards. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop: wave spawning, pathfinding, projectile and damage simulation.
  • Data-driven: towers, enemies, waves, and maps defined in data — ideal balancing case.
  • Custom tool: map + wave editor as the primary tool deliverable.
  • External DB: leaderboards. Auth ⏭️: optional. Analytics ⏭️: level-difficulty funnels.
  • 2D: map/tower/projectile rendering. Networking: optional (co-op mode). 3D: optional.
  • Replay System: replay of best runs attached to leaderboard entries.

5. Rhythm action game

Platform fit: Native mobile — touch taps on the reference phone (Pixel 8a/9a); the PC build is used for development and audio-latency tuning.
Front end: Note-highway renderer, hit feedback, custom GUI. Languages: C++, GLSL.
Back end: Audio-clock engine, input timing capture, replay system, score services. Languages: C++.
Backend scope: Local only.
  • Simulation loop: fixed-timestep loop synced to the audio clock — timing precision IS the game.
  • Data-driven: beatmaps (notes, timing windows) fully authored as data. Custom tool: beatmap editor.
  • External DB: scores, shared beatmaps. Auth ⏭️: optional. Analytics ⏭️: per-song accuracy stats.
  • 2D: note highway and hit-feedback animation. Networking / 3D: not needed.
  • Replay System: input-accurate replays verify scores and showcase perfect runs.
