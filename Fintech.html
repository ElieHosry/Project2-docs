Fintech

Scope

Products about financial data and money flows — market data and trading simulation, budgeting, payments and settlement sandboxes, fraud monitoring. Precision, high-frequency data handling and auditability are the signature engineering demands.

Examples

  1. Stock-trading simulator — live or replayed market data feed, C++ pricing/portfolio engine, real-time custom-rendered charts, leaderboards in Firebase. References:

    • TradingView Paper Trading
    • Paper trading tutorial (YouTube)

  2. Personal budget tracker with forecasting — C++ engine simulates future balances from spending patterns; custom chart renderer; category rules editor. References:

    • YNAB
    • PocketSmith cash flow forecasts

  3. Crypto/currency arbitrage sandbox — simulated exchange feeds at high frequency, order-book visualization, strategy configuration tool. References:

    • amCharts live order book / depth chart demo
    • Bitcoin order book visualization (YouTube) No single public product matches the full concept — illustrative mockup:

  4. Group expense splitter with settlement engine — C++ service computes optimal debt settlement across groups in real time; receipts and balances in Firebase; rule/currency configuration tool. References:

    • Splitwise
    • Splitwise settle-up tutorial (YouTube)

  5. Fraud detection & monitoring sandbox — simulated transaction streams scanned in real time by a C++ rules/anomaly engine; analyst dashboard with alert queue. References:

    • Stripe Radar
    • Stripe Radar analytics (docs)
    • Feedzai

How each requirement applies

Common to all five: C++ Performant Service (the financial computation engine) · Application Communication (UI ↔︎ engine) · Continuous Integration · Front-end + Local Backend · Data-Driven Architecture (products, rules, and fees as data) · Authentication ✅ (accounts are central to the category).

1. Stock-trading simulator

Platform fit: Native mobile + Web browser on mobile — traders check positions on the go.
Front end: Custom real-time charts, order entry & portfolio UI. Languages: HTML/JS/CSS + C++ chart renderer via WASM (or native C++).
Back end: Market feed engine, pricing/portfolio engine, transaction ledger, Firebase accounts. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): live/replayed market feed processed tick-by-tick; C++ pricing and portfolio engine.
  • 2D (adapted): custom-rendered real-time candlestick/line charts (no chart libraries).
  • Custom tool (adapted): market data replay/configuration tool.
  • External DB / Analytics ⏭️: portfolios, leaderboards, trading-behavior stats.
  • Ledger & audit trail: every simulated order/fill recorded double-entry — portfolio always reconciles.

2. Personal budget tracker with forecasting

Platform fit: Native mobile — expense entry happens in the moment.
Front end: Expense entry forms, projection charts, category UI. Languages: C++ (native mobile UI), GLSL.
Back end: Forecast simulation engine, budget ledger, cross-device sync. Languages: C++.
Backend scope: Local only.
  • Simulation loop (adapted): Monte-Carlo-style forward simulation of balances from spending patterns.
  • 2D (adapted): custom chart renderer for projections. Custom tool (adapted): category rules and scenario editor.
  • External DB / Analytics ⏭️: transactions, budgets, cross-device sync.
  • Ledger & audit trail: transactions stored as an immutable double-entry ledger; edits are corrections, not overwrites.

3. Crypto/currency arbitrage sandbox

Platform fit: Web browser on mobile — WASM dashboard with panels adapted to the phone layout; the PC build is the development vehicle.
Front end: Multi-panel dashboard, depth charts, strategy config UI (ImGui). Languages: C++, GLSL.
Back end: High-frequency feed engine, order-book maintenance, strategy engine, trade log. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop: genuinely high-frequency — simulated exchange feeds at 100+ Hz, order-book maintenance, opportunity detection. Closest fintech match to the raw requirement.
  • 2D (adapted): order-book depth charts and spread visualization. Custom tool (adapted): strategy/fees configuration tool.
  • External DB / Analytics ⏭️: session results, strategy performance history.
  • Ledger & audit trail: full trade log with fees per venue — enables honest P&L verification.

4. Group expense splitter with settlement engine

Platform fit: Native mobile + Web browser on mobile — group members join via a link.
Front end: Group/balance UI, debt-graph visualization, receipts. Languages: HTML/JS/CSS + WASM (or native C++).
Back end: Settlement optimization engine, expense ledger, Firebase groups & auth. Languages: C++.
Backend scope: Local only.
  • Simulation loop (adapted): real-time recomputation of optimal debt settlement graph as expenses change.
  • 2D (adapted): balance/debt-graph visualization. Custom tool (adapted): rules and currency configuration.
  • External DB / Analytics ⏭️: groups, receipts, balances; auth central for membership.
  • Ledger & audit trail: the settlement graph derives from an append-only expense ledger — who changed what, when.

5. Fraud detection & monitoring sandbox

Platform fit: Web browser on mobile + Native PC (analyst ops) — monitoring and alerts on the phone; the dense analyst review desk stays on the PC side.
Front end: Analyst dashboard, alert queue, anomaly timelines (ImGui). Languages: C++, GLSL.
Back end: Transaction stream scoring engine, rules engine, immutable case log. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop: high-frequency transaction stream scored rule-by-rule in real time — a direct fit.
  • Data-driven: fraud rules and thresholds as data. Custom tool (adapted): rule editor + case dashboard.
  • 2D (adapted): anomaly timelines and live metric rendering.
  • Import / alerts / security: CSV replay of historical data; alert queue; input validation and data protection.
  • Ledger & audit trail: flagged transactions and analyst decisions form an immutable case log.

1. Stock-trading simulator

M1:

  • G1 ⭐ Application Shell, Real-time C++ Simulation Loop, Input & Core Logic
    Tech & dependencies
    • Tech: SDL2 shell with a fixed-timestep C++17/20 loop that consumes market ticks and routes order-entry input; CMake, Git.
    • Prerequisites: None — foundation of this project.
    • Feeds into: M1 #4 (feed engine runs on its tick), M1 #11 (orders processed in the loop); every other goal.
  • G2 ⭐ Data-Driven Architecture of Core Objects / Entity System
    Tech & dependencies
    • Tech: Instruments, orders and portfolios defined as JSON data (nlohmann/json) with hot-reloadable instrument lists and market parameters.
    • Prerequisites: G1 (shell & loop).
    • Feeds into: M1 #11 (order/portfolio model), M2 G2 (replay/config editor edits this data), M2 #7 (ledger records these objects).
  • G3 ⭐ Debugging, Profiling, Stress Test & CI
    Tech & dependencies
    • Tech: ImGui overlay showing tick throughput and frame time, Tracy profiling of the tick→chart path, Catch2 tests on pricing math, stress replay at accelerated speed, GitHub Actions Windows+Linux.
    • Prerequisites: G1 (loop to instrument).
    • Feeds into: M2 G3 (CI advanced); safety net for M1 #4 and #11.
  • #4 Real-time Data Feed Engine
    Tech & dependencies
    • Tech: Producer/consumer threads pushing live or CSV-replayed market ticks through a lock-free ring buffer into the simulation loop.
    • Prerequisites: G1 (fixed-timestep loop as consumer).
    • Feeds into: M1 #5 (charts draw its ticks), M1 #11 (orders fill against its prices), M2 #4.
  • #5 Custom 2D Chart Rendering
    Tech & dependencies
    • Tech: Candlestick/line charts rendered from scratch with OpenGL/SDL2 and FreeType axis labels — no chart libraries; compiled to WASM via Emscripten for the mobile-web front-end.
    • Prerequisites: G1 (render loop); consumes M1 #4 ticks.
    • Feeds into: M2 #4 (advanced charting builds on this renderer).
  • #11 Order Processing Engine
    Tech & dependencies
    • Tech: Order validation and matching against the simulated market with fills priced in fixed-point money math, plus a per-order event log.
    • Prerequisites: M1 #4 (prices) + G2 (order/portfolio data model).
    • Feeds into: M2 #7 (every fill becomes a ledger entry), M2 #8 (portfolio results for leaderboards).

M2:

  • G1 ⭐ Architecture — Optimization & Platform Validation
    Tech & dependencies
    • Tech: Tracy profiling of tick-to-chart latency, ASan on MSVC, feed/render threading (std::thread); Emscripten web build validated in a mobile browser.

    • Feeds into: M2 #4 (GPU charting headroom), M2 #8 (stable client for sync).
  • G2 ⭐ Content Editor — Core
    Tech & dependencies

    • Libraries/Software: Dear ImGui, nlohmann/json

    • Prerequisites: M1 G3 (CI foundation).
    • Feeds into: Release/showcase builds of every M2 goal.
  • #4 Charting — Advanced
    Tech & dependencies
    • Tech: GPU-accelerated OpenGL chart rendering with indicators (SMA/EMA, volume), zoom/pan interaction and multi-timeframe views.
    • Prerequisites: M1 #5 (chart renderer) + M2 G1 (optimization).
    • Feeds into: Trader-facing showcase UI on desktop and mobile web.
  • #7 Ledger & Audit Trail — Complete
    Tech & dependencies
