Project Module — Introduction

Overview

A project module for second-year Computer Science students. Students form teams and build a project in a category of their choice. The project spans two trimesters; current planning focuses on the first trimester.

Target Platforms

  • Web browser on mobile (Android and iPhone devices)
  • Native mobile (Android — reference device: Google Pixel 8a / 9a; emulator available during development)
  • Web front-end (HTML, JS, CSS, WASM)
  • Server platform (Linux OS) — maybe in second trimester
  • Database, Authentication, Analytics (Firebase)

Native PC (Windows) is the development, debugging and first-testing platform for every team — it is not a shipping target. Each project's final output runs on Native mobile and/or Web browser on mobile; PC may ship only as the operator/authoring side of a role split (see Appendix F in the linked doc set).

Core Tech Requirements

  • External database (Firebase encouraged)
  • Authentication — maybe in second trimester
  • Analytics — maybe in second trimester
  • C++ performant service
  • Real-time high-frequency application: C++ simulation loop
  • Application communication
  • Continuous Integration
  • Data-driven architecture
  • Front-end + local backend (server backend maybe in second trimester)
  • Custom tools development for the product (e.g., scene editor if the project needs it)
  • Product does not need to be full-stack if tools are already full-stack (practicing full-stack is the point)
  • Graphical output mostly 2D; 3D optional and limited to strong students
  • Networking system allowed if the project needs it (e.g., multiplayer games)

Library Usage

  • Allowed: low-level libraries — OpenGL, SDL2, FMOD, OpenAL, ImGui, FreeType, etc.
  • Not allowed: high-end libraries or game engines

Project Categories

  • Games
  • Education + Training
  • Simulator
  • Fintech
  • GovTech
  • Social Media
  • CMS
  • Immersive Experience
  • Creative Tools & Media Production
  • Visualization & Analytics

First Trimester Timeline

Week Event
3 Teams formed
4 Project idea proposed; foundation sprint starts (head start on G1 shell/loop and CI — project-agnostic work)
5 Project approved and confirmed for kick-off; remaining goals finalized
9 Milestone 1 (Core technology base phase)
13 Milestone 2 (Core technology showcase phase)

Milestone 1 sits at week 9 (rather than 10) to balance the workload: with the week-4 foundation head start, both milestone phases carry roughly equal weekly effort (~10 h/week per 40-hour goal). See the Appendices for the full rationale.

Milestones & Goals

  • Each milestone has goals a team must work on (e.g., each member owns a goal).
  • A goal can be a high-level system (e.g., "Physics System") that a student breaks down into multiple tasks (~40 hours per milestone) (~10 hours per week, which include ~4 lab-hours).
  • Fairness: all teams get the same number of well-distributed goals per milestone, regardless of project category.
  • Team due diligence: instructors follow up with each team on a weekly basis (goal progress, blockers, integration health).

Goal Structure

  • Common goals (G1–G3 of each milestone): shared across all project categories.
  • Customizable goals (remaining 3): based on the team's chosen project category.
    • Teams within the same category may choose different customizable goals (instructor approval required).
    • Instructors maintain an initial pool of possible goals per project category (see CS Goals Pool per Category).
    • Teams may propose new goals for new project ideas.
