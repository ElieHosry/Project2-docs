Social Media

Scope

Multi-user social products — feeds, sharing, chat, communities, collaborative spaces. Real-time networking is the natural core, and a load-bearing C++ service (ranking, matching, shared state) is the guard-rail requirement: Firebase alone must never be the backend.

Examples

  1. Campus micro-blogging app — real-time feed and presence, C++ feed-ranking service, moderation dashboard as the custom tool. References:

    • Fizz
    • Yik Yak app overview (YouTube)

  2. Interest-based event meetup app — live chat per event, recommendation engine in C++, event-creation and moderation tools. References:

    • Meetup
    • Meetup tutorial (YouTube)

  3. Collaborative pixel-art canvas (r/place style) — shared real-time canvas, high-frequency update loop, rate-limiting service, admin/replay tool. References:

    • r/place (Wikipedia)
    • r/place 2022 timelapse (YouTube)

  4. Study-group matching & rooms app — students match by course/schedule, join live study rooms with shared timers and chat; C++ matching engine; moderation tools. References:

    • StudyStream focus room
    • Focusmate demo (YouTube)

  5. Campus photo sharing app — daily photo feed and galleries with a C++ media pipeline; moderation-first design. References:

    • BeReal (official)
    • How to use BeReal (YouTube)
    • BeReal (App Store)

How each requirement applies

Common to all five: Networking ✅ (real-time communication is the category's defining requirement) · C++ Performant Service (feed/matching/canvas engine) · Application Communication · Continuous Integration · Front-end + Local Backend · Data-Driven Architecture (feed rules, event types, room configs as data) · Authentication ✅ (identity is central).

1. Campus micro-blogging

Platform fit: Native mobile — feed apps live on the phone.
Front end: Feed UI, post composer, profile screens. Languages: C++ (custom UI) or HTML/JS/CSS.
Back end: Feed-ranking service, social graph, moderation queue, Firebase identity & data. Languages: C++.
Backend scope: Server-indicated (T2).
  • Simulation loop (adapted): C++ feed-ranking service scoring posts in real time.
  • 2D (adapted): custom feed UI rendering. Custom tool (adapted): moderation dashboard.
  • External DB / Analytics ⏭️: posts, follows, engagement metrics.
  • Moderation pipeline: word filters + report queue feeding the moderation dashboard — essential for a campus deployment.

2. Event meetup

Platform fit: Native mobile + Web browser on mobile — event links must open in the browser.
Front end: Event pages, per-event chat UI, discovery screens. Languages: HTML/JS/CSS (+ WASM).
Back end: Recommendation engine, chat backend (WebSockets), moderation, Firebase. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): recommendation engine matching users to events.
  • Networking: live per-event chat. Custom tool (adapted): event creation + moderation tools.
  • External DB / Analytics ⏭️: events, RSVPs, attendance stats.
  • Moderation pipeline: event/report review before promotion; chat message filtering.

3. Collaborative pixel canvas

Platform fit: Web browser on mobile + Native mobile — the shared canvas should be reachable from any phone; web-first. The PC build is used for development.
Front end: Shared canvas renderer, palette & zoom UI. Languages: HTML/JS/CSS + C++ canvas renderer via WASM, GLSL.
Back end: Canvas state engine (high-frequency ticks), rate limiter, sync server, history storage. Languages: C++.
Backend scope: Server-indicated (T2).
  • Simulation loop: genuinely high-frequency shared canvas state with rate limiting — the closest social match to the raw requirement.
  • 2D: canvas rendering applies directly. Custom tool (adapted): admin/replay (timelapse) tool.
  • External DB / Analytics ⏭️: canvas history, placement stats.
  • Moderation pipeline: automated pattern detection + admin rollback of vandalized regions.

4. Study-group matching & rooms

Platform fit: Web browser on mobile — join and run study sessions from the phone; the PC build is used for development.
Front end: Study room UI, shared timers, presence indicators. Languages: HTML/JS/CSS.
Back end: Matching engine, room/timer synchronization, chat backend, Firebase. Languages: C++.
Backend scope: Server-indicated (T2).
  • Simulation loop (adapted): matching engine (course/schedule) + synchronized room timers.
  • Networking: live rooms, presence, chat. Custom tool (adapted): moderation and room management.
  • External DB / Analytics ⏭️: courses, matches, session/focus statistics.
  • Moderation pipeline: room reports, user blocking, and chat filtering keep rooms safe.

5. Campus photo sharing app

Platform fit: Native mobile — a camera-first experience.
Front end: Camera capture flow, gallery/feed rendering. Languages: C++ (native mobile UI), GLSL.
Back end: Media pipeline (resize, filters, thumbnails), feed assembly, moderation queue, Firebase. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): feed assembly plus C++ image-processing pipeline (resize, filters, thumbnails).
  • Networking: real-time feed updates and presence.
  • Data-driven: post/gallery models as data. Custom tool (adapted): moderation dashboard.
  • 2D (adapted): custom gallery/feed rendering. External DB / Analytics ⏭️: media metadata, engagement events.
  • Moderation pipeline: image report queue and reviewer actions — the design center of this example.

1. Campus micro-blogging app

M1:

  • G1 ⭐ Application Shell, Real-time C++ Simulation Loop, Input & Core Logic
    Tech & dependencies
    • Tech: SDL2 shell with a fixed-timestep C++17/20 loop driving feed refresh, presence updates and the post composer's input handling; CMake, Git.
    • Prerequisites: None — foundation of this project.
    • Feeds into: M1 #4 (networking runs on its tick) and M1 #9 (ranking passes scheduled by the loop); every other goal.
  • G2 ⭐ Data-Driven Architecture of Core Objects / Entity System
    Tech & dependencies
    • Tech: Posts, users and feed-ranking rules as hot-reloadable JSON models (nlohmann/json), so ranking weights and post types are tunable data.
    • Prerequisites: G1 (shell and loop).
    • Feeds into: M1 #9 and #14 consume the post/user model; M2 G2 and #6 edit and moderate it.
  • G3 ⭐ Debugging, Profiling, Stress Test & CI
    Tech & dependencies
    • Tech: ImGui overlays showing live rank scores and socket traffic, Tracy profiling of ranking passes, Catch2 tests, stress runs with thousands of synthetic posts, GitHub Actions (Windows + Linux).
    • Prerequisites: G1.
    • Feeds into: M2 G3; the safety net proving M1 #4 and #9 hold under load.
  • #4 Real-time Networking Foundation
    Tech & dependencies
    • Tech: Winsock2/ENet client-server layer (native-mobile fit) with a custom serialized protocol for post publish/subscribe and presence heartbeats.
    • Prerequisites: G1 (runs on its tick).
    • Feeds into: M1 #14 (live follow/profile updates); M2 #6 (report submission) and real-time feed delivery.
  • #9 Ranking Engine — Foundation
    Tech & dependencies
    • Tech: The C++ feed-ranking service — weighted scoring of recency and engagement with priority queues and incremental re-ranking each tick.
    • Prerequisites: G1 (tick) and G2 (post/user model).
    • Feeds into: M2 #8 (recommendation layer); the ranked feed every user sees.
  • #14 Profile & Social Graph
    Tech & dependencies
    • Tech: Follows and profiles as Firestore adjacency lists with denormalized follower counts, surfaced in the custom C++ profile screens.
    • Prerequisites: G2 (user model); M1 #4 for live follow updates.
    • Feeds into: M1 #9 (follow graph as ranking input); M2 #7 attaches full Auth identity.

M2:

  • G1 ⭐ Architecture — Optimization & Platform Validation
    Tech & dependencies
    • Tech: Tracy/ASan pass over the ranking service and socket layer, feed scoring moved to worker threads, validation of the native-mobile target (Android emulator / Emscripten check).
    • Prerequisites: All M1 goals (hardens the M1 codebase).
    • Feeds into: M2 #8 (performance headroom for recommendations); packaged builds via G3.
  • G2 ⭐ Content Editor — Core
    Tech & dependencies
    • Tech: Admin tooling core in Dear ImGui — browse, edit and remove posts and users with JSON round-trip and an undo/redo command stack.
    • Libraries/Software: Dear ImGui, nlohmann/json
    • Prerequisites: M1 G2 (data-driven post/user model).
    • Feeds into: M2 #6 — the moderation dashboard is built directly on this core.
  • G3 ⭐ CI — CMake, Code Quality, Build Standards & Packaging
    Tech & dependencies
    • Tech: CMake presets, clang-format + clang-tidy, CPack packaging of the mobile/desktop clients, GitHub Actions matrix builds.
    • Prerequisites: M1 G3 (CI foundation).
    • Feeds into: Release/showcase builds of every goal.
  • #6 Moderation Dashboard & Pipeline
    Tech & dependencies
    • Tech: C++ wordlist/pattern filters on incoming posts plus a Firestore report queue surfaced in the G2 admin tool — essential for a campus deployment.
    • Prerequisites: M2 G2 (tool core), M1 G2 (content model), M2 #7 (identity behind reports).
    • Feeds into: Safe launch of the showcase feed.
  • #7 Firebase — Auth, Push & Analytics
    Tech & dependencies
    • Tech: Firebase Auth with campus email, FCM push for mentions and replies, Analytics on engagement events — REST API from C++.
    • Prerequisites: M1 G2 (data model to sync) and M1 #14 (graph already in Firestore).
    • Feeds into: M2 #6 (verified reporters) and M2 #8 (engagement signals).
  • #8 Ranking Engine II
    Tech & dependencies
    • Tech: Recommendation layer over the M1 ranker — weighted features and cosine similarity across the follow graph and engagement history, with candidate generation and evaluation metrics.
    • Prerequisites: M1 #9 and #14; M2 G1 (optimization headroom).
    • Feeds into: The showcase "for you" feed and discovery quality.
