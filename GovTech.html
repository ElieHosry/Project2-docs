GovTech

Scope

Institutional service platforms — for the public sector and private institutions alike (city services, universities, hospitals, companies). The shape that matters: a citizen/customer/member-facing side plus a staff/operations side, connected by service workflows, live data feeds and dashboards, with accountability (roles and audit trails) as the signature requirement. A campus attendance system belongs here as much as a city permit office.

Examples

  1. Queue management for public services — simulates and tracks citizen queues (license office, clinic); real-time dashboard; service-flow editor. References:

    • Qminder for healthcare
    • Qmatic queue system demo (YouTube)

  2. City resource dashboard — real-time simulated feeds (water, power, waste) processed in C++; map-based 2D visualization; sensor/zone configuration tool. References:

    • Snap4City control room dashboards
    • Snap4City dashboard demo (YouTube)

  3. Digital form & permit workflow — citizens submit applications, officers process them; data-driven form builder as the custom tool; Firebase auth for roles. References:

    • Cloudpermit building permitting
    • Typeform form builder

  4. Public transport tracker — simulated bus/train positions processed in a C++ feed engine; live 2D map with arrival predictions; route/timetable editor. References:

    • Transit app
    • Citymapper overview (YouTube)

  5. Citizen issue reporting (311-style) — citizens photo-report potholes and broken lights; staff triage dashboard; category workflows; public transparency stats. References:

    • SeeClickFix
    • Reporting an issue on SeeClickFix (YouTube)
    • FixMyStreet

How each requirement applies

Common to all five: C++ Performant Service (feed/queue/workflow engine) · Application Communication (UI ↔︎ engine) · Continuous Integration · Front-end + Local Backend · Data-Driven Architecture (services, forms, routes as data) · Authentication ✅ (citizen/staff roles are central).

1. Queue management

Platform fit: Web browser on mobile + Native PC (staff ops) — web browser on mobile for citizens (take a ticket), PC dashboard on the staff side.
Front end: Citizen ticket page (web), staff queue dashboard. Languages: HTML/JS/CSS (citizen); C++ + ImGui (staff).
Back end: Queue dynamics simulation, wait-time statistics, notifications, Firebase records. Languages: C++.
Backend scope: Server-indicated (T2).
  • Simulation loop: simulates citizen arrivals, service times, and queue dynamics in real time — a true simulation.
  • Custom tool (adapted): service-flow editor (counters, services, priorities). 2D (adapted): live queue dashboard rendering.
  • External DB / Analytics ⏭️: tickets, wait-time statistics; auth for staff vs. citizen roles.
  • RBAC & audit logging: counter staff vs. supervisor permissions; every ticket action logged.

2. City resource dashboard

Platform fit: Web browser on mobile + Native PC (control-room ops) — citizens and officials view the live dashboard on phones; the control-room display stays on the PC side.
Front end: Map layers, live dashboard panels, alert displays. Languages: C++, GLSL.
Back end: Sensor feed engine, aggregation & statistics, zone configuration, Firebase history. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): C++ engine generates and processes simulated sensor feeds (water, power, waste).
  • 2D (adapted): map-based visualization with live layers. Custom tool (adapted): sensor/zone configuration tool.
  • External DB / Analytics ⏭️: historical feed data, alert statistics.
  • RBAC & audit logging: operator vs. admin zones; configuration changes audited.

3. Digital form & permit workflow

Platform fit: Web browser on mobile (citizens apply) + Native PC (officer ops) (officers review) — a natural two-front-end split.
Front end: Citizen forms rendered from JSON schemas, officer review UI. Languages: HTML/JS/CSS (citizen); C++ + ImGui or web (officer).
Back end: Workflow/state machine engine, validation engine, RBAC & audit, Firebase. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): weakest fit — real-time workflow/validation engine; teams should scope this consciously.
  • Custom tool: the data-driven form builder applies directly and is the strongest deliverable here.
  • External DB / Analytics ⏭️: applications, statuses, processing-time metrics; auth central (citizen/officer roles).
  • RBAC & audit logging: the defining requirement here — who approved which permit, full decision trail.

4. Public transport tracker

Platform fit: Native mobile + Web browser on mobile — commuters check arrivals on the phone.
Front end: Live transit map, arrival boards, route views. Languages: HTML/JS/CSS + WASM map renderer (or native C++).
Back end: Vehicle movement simulation, arrival prediction, timetable services, Firebase. Languages: C++.
Backend scope: Server-indicated (T2).
  • Simulation loop: simulates vehicle movement along routes and computes arrival predictions — a true simulation.
  • 2D (adapted): live map rendering with vehicle positions. Custom tool (adapted): route/timetable editor.
  • External DB / Analytics ⏭️: routes, timetables, ridership stats.
  • RBAC & audit logging: operators edit routes/timetables under role permissions with change history.

5. Citizen issue reporting (311-style)

Platform fit: Native mobile + Native PC (staff ops) — Native mobile for citizens (photo + location), PC on the staff-triage side.
Front end: Citizen report flow (camera, map pin), staff triage board. Languages: C++ (native mobile) or HTML/JS/CSS; C++ + ImGui (staff).
Back end: Report feed processing, deduplication & priority scoring, RBAC & audit log, Firebase. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop (adapted): real-time report feed processing, deduplication, priority scoring.
  • Data-driven: issue categories and triage workflows as data. Custom tool (adapted): staff dashboard + category editor.
  • 2D (adapted): map rendering with issue pins and heat layers.
  • Validation / notifications: photo and location validation; status-change notifications to reporters.
  • RBAC & audit logging: citizen vs. staff vs. supervisor roles; every triage decision logged.

1. Queue management for public services

M1:

  • G1 ⭐ Application Shell, Real-time C++ Simulation Loop, Input & Core Logic
    Tech & dependencies
    • Tech: SDL2/ImGui shell, fixed-timestep loop simulating citizen arrivals and service times; CMake, Git.
    • Prerequisites: None — foundation of this project.
    • Feeds into: M1 G4 (queue engine runs on its tick); every other goal.
  • G2 ⭐ Data-Driven Architecture of Core Objects / Entity System
    Tech & dependencies
    • Tech: Counters, services, priorities, and ticket types defined as JSON entities (nlohmann/json) with hot-reload.
    • Prerequisites: G1 (shell and loop to host the entity system).
    • Feeds into: M1 G4 (engine consumes service definitions); M2 G2 (service-flow editor edits this data).
  • G3 ⭐ Debugging, Profiling, Stress Test & CI
    Tech & dependencies
    • Tech: ImGui overlay of live queue state, Tracy profiling, Catch2 stress tests flooding the queue with thousands of simulated citizens, GitHub Actions Windows/Linux builds.
    • Prerequisites: G1 (running loop to instrument).
    • Feeds into: M2 G3 (CI hardening); validates M1 G4 under load.
  • G4 Real-time Feed / Queue Engine
    Tech & dependencies
    • Tech: C++ producer/consumer threads generating ticket events (issue, call, serve, no-show) through a lock-free ring buffer, with CSV replay of past office days.
    • Prerequisites: G1 (simulation tick); G2 (service and counter definitions).
    • Feeds into: M1 G11 (event stream for statistics), M1 G12 (ticket-called triggers), M2 G4 and M2 G10.
  • G11 Statistics & Wait-Time Analytics
    Tech & dependencies
    • Tech: Per-service/per-counter wait and service-time aggregation in C++, custom SDL2/OpenGL charts (no chart libs), history persisted to Firestore.
    • Prerequisites: M1 G4 (ticket event stream); G2 (event data model).
    • Feeds into: M2 G10 (historical calibration for predictions); M2 G4 (dashboard heatmaps and history).
  • G12 Notification System
    Tech & dependencies
    • Tech: Firebase Cloud Messaging push for "your ticket is called", in-app event bus for the staff dashboard, service worker on the citizen web ticket page.
    • Prerequisites: M1 G4 (queue events to notify on); Firebase project with citizen/staff Auth.
    • Feeds into: Citizen ticket experience at showcase; M2 G10 (pushing predicted wait times).

M2:

  • G1 ⭐ Architecture — Optimization & Platform Validation
    Tech & dependencies
    • Tech: Tracy profiling of the queue engine under thousands of concurrent tickets, ASan/UBSan, memory pools and threading tuning, Emscripten validation of the citizen web page.
    • Prerequisites: All M1 goals (hardens the M1 queue codebase).
    • Feeds into: M2 G4 (dashboard frame budget) and M2 G10 (prediction throughput).
  • G2 ⭐ Content Editor — Core
    Tech & dependencies
    • Tech: Service-flow editor in C++/ImGui — configure counters, services, and priority rules with JSON round-trip and an undo/redo command stack.
    • Libraries/Software: Dear ImGui, nlohmann/json
    • Prerequisites: M1 G2 (service/counter data model it edits).
    • Feeds into: M1 G4 (engine runs edited flows live); M2 G4 (dashboard reflects reconfigured offices).
  • G3 ⭐ CI — CMake, Code Quality, Build Standards & Packaging
    Tech & dependencies
    • Tech: CMake presets, clang-format/clang-tidy gates, CPack packaging of the staff dashboard plus Emscripten citizen web build in the Actions matrix.
    • Prerequisites: M1 G3 (CI and test foundation).
    • Feeds into: Release builds of every goal for the showcase.
  • G4 Dashboard — Advanced
    Tech & dependencies
    • Tech: Supervisor dashboard with live queue layers per counter, wait-time heatmaps, and scrubbing through the day's queue history (SDL2/OpenGL + ImGui).
    • Prerequisites: M1 G4 (live events) and M1 G11 (aggregates/history); M2 G1 (rendering budget).
    • Feeds into: Staff-side showcase; M2 G7 gates its supervisor-only views.
  • G7 RBAC & Audit Logging
    Tech & dependencies
    • Tech: Firebase Auth custom claims for counter staff vs. supervisor, permission checks in the C++ service, append-only audit log of every ticket action (call, skip, transfer, close).
    • Prerequisites: M1 G4 (ticket actions to log); M1 G11/G12 (Firebase records and identity in place).
    • Feeds into: M2 G4 (role-gated dashboard panels); trust story of the whole product.
  • G10 Prediction Engine (wait times)
    Tech & dependencies
    • Tech: Exponential smoothing / simple Kalman filtering over queue lengths and service rates, calibrated per service against M1 wait-time history.
    • Prerequisites: M1 G4 (live queue state) and M1 G11 (historical statistics).
    • Feeds into: Citizen ticket page ETAs, M1 G12 (predicted-wait notifications), M2 G4 (forecast panel).
