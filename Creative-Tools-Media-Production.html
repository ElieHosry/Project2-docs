Creative Tools & Media Production

Scope

Professional creation tools — paint and animation editors, audio editors, trackers, VFX authoring. The tool IS the product and its users are creators; editing engines, live preview and non-destructive workflows are the signature engineering.

Examples

  1. Pixel-art & animation editor — layered canvas painting, palette tools, onion-skin animation, sprite-sheet/GIF export; the tool IS the product. References:
    • Aseprite (official)
    • Aseprite docs — layers, animation, export
  2. Audio editor / mini-DAW — waveform display and editing, multitrack mixing, real-time effect processing; the DSP callback is a true high-frequency loop. References:
    • Audacity (official)
    • ocenaudio (official)
  3. Music tracker / chiptune studio — pattern-based sequencer driving a real-time synthesizer engine (square/wave/noise channels), instrument editor, module export. References:
    • FamiTracker (official)
    • Little Sound Dj (official)
  4. Collaborative diagram & vector editor — shapes, bézier paths, snapping and alignment, live multi-user collaboration; SVG import/export. References:
    • Excalidraw (official)
    • Inkscape (official)
  5. Particle & VFX editor — author particle systems with live GPU preview; bake flipbooks/sprite sheets and export JSON assets consumable by other teams' projects. References:
    • EmberGen (JangaFX)
    • EmberGen documentation

How each requirement applies

Common to all five: Custom Tools ✅ (the tool IS the product, satisfying the "tools already full-stack" clause) · C++ Performant Service (the editing/preview engine itself) · Data-Driven Architecture ✅ (documents, brushes, instruments, emitters as data) · Application Communication (UI ↔︎ C++ engine messaging) · Continuous Integration · Front-end + Local Backend.

1. Pixel-art & animation editor

Platform fit: Native mobile — touch pixel-art painting on the reference phone (Pixel 8a/9a) (zoomed-grid workflow); the PC build is the development/iteration vehicle.
Front end: Canvas renderer (OpenGL), layer & frame timeline UI, palette and tool panels. Languages: C++, GLSL.
Back end: Document engine (layers, blend modes, command graph), exporters (PNG/GIF/sprite sheet), Firebase palette/asset sharing. Languages: C++.
Backend scope: Local only.
  • Simulation loop: real-time canvas compositing and animation preview — layers, blend modes and onion skin re-composited every frame.
  • Data-driven: documents, palettes, brushes and export presets fully described as data.
  • Custom tool: the editor IS the product. 2D: pixel-perfect canvas rendering, blend modes, grid/tiled preview.
  • External DB: shared palette/asset library. Auth ⏭️: optional accounts. Analytics ⏭️: feature-usage stats.
  • Networking / 3D: not needed — effort goes into tool quality.
  • Non-destructive edit history: command graph with unlimited undo, history panel and crash recovery (Aseprite-style data recovery).

2. Audio editor / mini-DAW

Platform fit: Native mobile — simplified touch UI (waveform, transport, mixer) on the reference phone (Pixel 8a/9a); AAudio provides the low-latency path. PC build for development and heavy-file tests.
Front end: Waveform/spectrogram renderer, multitrack timeline, mixer & effect-rack UI. Languages: C++, GLSL.
Back end: DSP engine (mixing, effects, resampling), codec import/export, project persistence, Firebase project sync. Languages: C++.
Backend scope: Local only.
  • Simulation loop: the real-time audio callback — mixing and effect chains processed at audio rate, the tightest timing budget in the catalog.
  • 2D: waveform and spectrogram rendering with smooth zoom from hours down to single samples.
  • Data-driven: projects, tracks, effect chains and presets as data. Custom tool: the editor IS the product.
  • External DB: cloud project/preset sync. Auth ⏭️: optional. Analytics ⏭️: effect-usage stats.
  • Networking / 3D: not needed.
  • Non-destructive edit history: edits stored as a command list over immutable source audio — the professional editing pattern; full session recovery.

3. Music tracker / chiptune studio

Platform fit: Native mobile + Native PC (authoring) — playback and live performance on the phone; keyboard-driven pattern authoring on the PC side.
Front end: Pattern-grid editor (custom widgets), instrument editor, oscilloscope/VU views. Languages: C++, GLSL.
Back end: Synthesizer engine (channels, envelopes, arpeggios), sample-accurate sequencer clock, module serialization, Firebase song library. Languages: C++.
Backend scope: Local only.
  • Simulation loop: sample-accurate sequencer driving a real-time synth engine — audio-rate DSP with musical timing precision.
  • Data-driven: instruments, patterns and songs as pure data; module format designed by the team.
  • Custom tool: the tracker IS the product. 2D: pattern grid, oscilloscopes and spectrum views.
  • External DB: shared song/instrument library. Auth ⏭️: optional. Analytics ⏭️: most-used instruments/effects.
  • Networking / 3D: not needed.
  • Non-destructive edit history: per-pattern edit history with named checkpoints and autosave/session recovery.

4. Collaborative diagram & vector editor

Platform fit: Web browser on mobile — collaboration is web-first (WASM build); the PC build is used for development and precision-editing tests.
Front end: Vector canvas renderer (bézier tessellation, snapping, selection), shape/connector tools. Languages: C++, GLSL (via WASM); HTML/JS/CSS shell.
Back end: Document/scene-graph model, operation-log sync service for live collaboration, SVG import/export, Firebase documents & sharing. Languages: C++.
Backend scope: Server-indicated (T2).
  • Simulation loop (adapted): interactive render loop — tessellation, snapping and live remote cursors updated every frame.
  • Networking: core requirement — real-time collaborative editing over WebSockets with operation sync and conflict resolution.
  • Auth: applies directly — documents, owners and share links. External DB: document storage. Analytics ⏭️: collaboration stats.
  • Data-driven: the document is a pure operation log / scene graph. Custom tool: the editor IS the product.
  • 2D: bézier rasterization, hit-testing, snapping guides. 3D: not needed.
  • Non-destructive edit history: the history doubles as the collaboration operation log — undo/redo that works correctly with multiple simultaneous editors.

5. Particle & VFX editor

Platform fit: Native mobile — authoring with simplified panels and live GPU preview on the reference phone (Pixel 8a/9a); PC build for development and heavy bakes.
Front end: Live preview viewport (OpenGL instancing), emitter/curve editors, preset browser. Languages: C++, GLSL.
Back end: Particle simulation core, flipbook/sprite-sheet baker, JSON asset exporter, Firebase preset library. Languages: C++.
Backend scope: Local now, server-ready (T2).
  • Simulation loop: the particle simulation IS the loop — thousands of particles with forces, collisions and emission curves per frame.
  • Data-driven: emitters, forces and curves as JSON — the exported assets are consumable by other teams' projects (a natural cross-team showcase).
  • Custom tool: the editor IS the product. 2D: instanced rendering, additive blending, flipbook baking. 3D: optional 2.5D particles for strong students.
  • External DB: shared preset library. Auth ⏭️: optional. Analytics ⏭️: preset popularity.
  • Networking: not needed.
  • Non-destructive edit history: preset history with branching — author variants of an effect and compare them side by side.
