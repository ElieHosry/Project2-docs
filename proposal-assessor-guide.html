Using the Project Proposal Assessor

A guide for instructors. The assessor is a Claude Cowork skill (Project-Proposal-Assessor.skill, stored in the project root) that evaluates a student project proposal against this module's 9 categories and produces the full assessment package automatically. It is advisory — the instructor remains the approval gate at week 5.

What you need

  • The Claude desktop app (Cowork mode). The skill does not work on claude.ai in the browser.
  • Ideally, this project folder connected in Cowork — the skill then reads the live category pages, goal pools and matrix, and saves its outputs into Student Projects/. Without the folder it still works from bundled snapshots (see Standalone use below).

Installing

  1. Get the file Project-Proposal-Assessor.skill from the project root (or from a colleague).
  2. Drop it into a Cowork chat — it appears as a card with a Save skill button.
  3. Click Save skill. Manage or remove it later under Settings → Capabilities.

To update to a newer version: remove the installed skill, then save the newer file the same way.

Assessing a proposal

Connect the project folder, then type:

assess this proposal: <the student's few lines describing the idea>

Give a project name if you have one; otherwise the assessor will ask. It then:

  1. Checks module fit — can a real-time C++ loop be load-bearing? real 2D work? a meaningful custom tool? no Firebase-swallows-everything?
  2. Applies the category gate — the idea must fit one of the 9 categories; new categories are never invented.
  3. Issues a three-tier verdict: ✅ Fit · 🔶 Fit with adjustments (listed, and to be treated as approval conditions) · ❌ Reject — distinguishing "fails the module" from "fits the module but no category", the latter with a reshaping hint for resubmission.
  4. Web-verifies 3 references to comparable real products (never invented links).
  5. Suggests M1/M2 goals: G1–G3 ⭐ fixed plus 3 picks from the matched category's pool, each with a Tech & dependencies block tailored to the proposal.

What it produces

For accepted (✅/🔶) proposals, under Student Projects/<Category>/:

  • <project-name>.md — the assessment page: verdict banner, proposal, adjustments, references, one-column Core Tech Requirements table (with the category's purple requirement), platform/front-end/back-end boxes with backend scope, and the suggested goals.
  • <project-name>-goals/ — the two milestone goal workbooks (M1_…​.xlsx, M2_…​.xlsx), derived from the category templates in Sample Templates/Goals/, with tailored acceptance criteria (2–5 bullets per goal), Team Info prefilled, and the Owner column left blank for the team.
  • Rebuilt Student Projects/index.md and the self-contained HTML set in Student Projects/html/.

Rejected proposals are logged in Student Projects/rejected-log.md (reason, rejection type, nearest category, resubmission hint) — no assessment file is created.

Standalone use (colleagues without this folder)

The skill carries snapshots of the grounding material (category rubric, goal-pool digest, requirements matrix, appendices, stylesheet, template workbooks), so it can assess proposals with no folder connected — outputs are offered as downloadable files instead. Two limitations: back-links into this doc set's HTML will be dead, and the snapshots reflect the doc set as of the last packaging.

Using it outside Claude (other AI agents)

The skill follows the open SKILL.md convention (YAML frontmatter + markdown instructions + scripts/ and reference/ directories), which as of 2026 is also supported by OpenAI's Codex CLI, Google's Gemini CLI, GitHub Copilot's agent mode, Cursor, and community tools such as Cline and Windsurf. This skill uses only the core format, so it works across those tools without modification. Practical notes:

  • Packaging: the one-click Save-skill install is a Claude-apps feature. For other tools, unzip the .skill file (it is an ordinary zip) and place the project-proposal-assessor/ folder in that tool's skills directory.
  • Capabilities assumed: web search/fetch (reference verification), a shell with Python (openpyxl) and pandoc (workbooks and HTML), and write access to the project folder. The instructions tell the agent to skip and report any step its harness cannot perform.
  • Phrasing: a few instructions are worded for Claude's desktop app ("connected folder", "offer the file for download"); capable agents interpret these loosely.
  • Quality travels with the model, not the file. The mechanics port cleanly, but the judgment calls — category gate, three-tier verdict, tailored goals — are only as good as the model running the skill. Spot-check assessments produced by other agents before trusting them.

Reference: the Agent Skills specification.

Maintenance

  • After major doc changes (goal pools, categories, appendices, or the goal templates), ask Claude to re-package the skill so the bundled snapshots stay current — then re-save it and redistribute to colleagues.
  • If the skill errors about a truncated or corrupted script: the installed copy got damaged during caching. Remove the skill and re-save it from a fresh copy of the file. (The skill also carries self-healing instructions and will usually reconstruct the script on its own.)
  • The .skill file is an ordinary zip — safe to email or share however you like.
