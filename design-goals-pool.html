Design Goals Pool

Status: approved by the Design instructor (July 2026). Distilled from the previous module's Designers-Rubrics (M1–M3) and the New Preliminary Design timeline. The base pool below is Games-flavored; the Category adaptations section covers all 9 categories. Amendment (July 9): the audio picks (former M1 #9, M2 #10) migrated to the TEAM Goals; the pools below are renumbered.

How Design goals work

Each team has 1–2 Design students (minimum 1, mandated). Unlike the CS track (one 40 h goal per student per milestone), Design goals are lighter, deliverable-sized units: each Design student takes 2 goals per milestone (two Design goals ≈ one CS goal in effort). The mandated designer owns the common goal D1 ⭐ plus 1 pool pick; a second designer takes 2 more picks (all picks with instructor approval).

Default picks: the mandated designer's M1 pick is normally #2 Core-Loop Interactive Prototype — this preserves the old module's compulsory GDD + prototype pair on every team.

Conditional rule: pool goals marked (conditional) are mandatory picks for teams with no Art student. Since this consumes a pick, teams with 0 artists are strongly advised to carry 2 designers; a solo-designer/0-artist team must ask the instructor which conditional bends (in that case only, D1 ⭐ absorbs a minimal core-loop sketch in place of #2).

Sizing: Design goals are sized by deliverable + quality tier, not by a strict 40-hour estimate. Every goal carries a Quality ladder (Base → Good → Exceptional) distilled from the old rubric tiers (Simple/Moderate → Good/Advanced → Exceptional). The Base rung is what "Done" means; upper rungs raise the grade, they don't extend the deadline.

Numbering: D1 ⭐ = common; #2, #3, … = pool picks. Pools are extensible like the CS pools — instructor-approved goals from new proposals are appended at the end.

Cross-track dependencies: several goals reference CS goals (G1–G3 ⭐, #N) from the Games pools — Design deliverables feed the CS data-driven architecture and editors, and consume the CS prototype builds.

Not in this pool: audio direction/production and the team-wide UI baseline live in the TEAM Goals — Audio & UI (TA ⭐ / TU ⭐ — mandatory for every team, shared grade, graded by the Audio/UI instructors).

Design — Games

Milestone 1 (Research phase)

  1. ⭐ Game Design Document, Problem Statement & Feasibility — the design backbone: concept, core loop, mechanics, USP, scope sign-off.
    Details & dependencies
    • Deliverable: 6–8 page GDD (core mechanics, objective, theme, genre, core game loop, control mapping incl. touch, level/progression overview, references) + the team's problem statement + a feasibility check against the CS engine feature plan, signed off by all members + a trimester planning timeline. (Solo-designer/0-artist teams only: may absorb a minimal core-loop sketch in place of #2 — see the conditional rule above.)
    • Prerequisites: None — foundation of the design track.
    • Feeds into: Every goal in both tracks; the wk4 product pitch; CS M1 G2 (what the core objects ARE); M2 D1 ⭐.
    • Quality ladder: Base = complete, clear, feasible. Good = systems interlock with no purposeless mechanics, depth over volume, clean visuals-supported writing. Exceptional = genuinely unique USP that moves the genre forward.
  2. Core-Loop Interactive Prototype — playable proof of the core mechanics in Figma or a commercial engine; the design risk killer.
    Details & dependencies
    • Deliverable: Playable prototype without critical crashes; title/help screen listing controls; readme (controls, how to play, feature list); touch controls follow industry standards.
    • Prerequisites: D1 ⭐ (core loop defined).
    • Feeds into: M1 #9 playtests; the wk9 prototype checkpoint; CS team's target reference for M2.
    • Quality ladder: Base = core mechanics and player interaction are evident. Good = industry-standard minimal controls with audio-visual feedback that explains the moment. Exceptional = the USP is already felt in play, not just described.
  3. User Research, Personas & Competitive Analysis — know the player and the genre before betting a trimester on them.
    Details & dependencies
    • Deliverable: Competitor/genre teardown (3+ titles), target-player personas, research questions with method and findings; implications written back into the GDD.
    • Prerequisites: None — runs from wk3 alongside team formation.
    • Feeds into: D1 ⭐ (audience-fit decisions), M1 #4 (journey-driven flows), M2 #5 (recruiting playtesters that match personas).
    • Quality ladder: Base = personas + teardown exist and are referenced. Good = findings visibly changed design decisions (documented before/after). Exceptional = original research (surveys/interviews) with actionable insight beyond genre convention.
  4. UX Flows & Wireframes — Menus → HUD — the screen map of the whole product before pixels get expensive.
    Details & dependencies
    • Deliverable: Complete screen inventory and navigation map (boot → menus → gameplay → fail/success loops), wireframes for all critical screens, HUD information hierarchy, user journeys per persona.
    • Prerequisites: D1 ⭐ (what screens the game needs).
    • Feeds into: M2 D1 ⭐ (polished UI), M2 #3 (motion design), CS M2 #5 (custom GUI system scope).
    • Quality ladder: Base = all critical screens wireframed and connected. Good = information hierarchy justified; journeys traceable per persona. Exceptional = wireframes usability-tested and iterated before M1 ends.
  5. Control Mapping & Game Feel Specification — touch-first input design and the feel rules that make it responsive.
    Details & dependencies
    • Deliverable: Input schemes for the mobile target (touch zones, gestures) plus keyboard/controller dev fallbacks; game-feel spec (buffering, acceleration curves, forgiveness mechanics) as tunable parameters.
    • Prerequisites: D1 ⭐ (mechanics to map).
    • Feeds into: CS M1 G1 ⭐ (input system requirements), M2 #8 (juice pass), M2 #10 (ergonomics).
    • Quality ladder: Base = complete, standards-compliant mapping. Good = feel parameters specified as data the CS team can tune. Exceptional = feel validated in the prototype with test evidence.
  6. Level & Progression Design — the difficulty curve and the levels that teach the game without a manual.
    Details & dependencies
    • Deliverable: Level design document: paper maps/greyboxes, difficulty and pacing curve, teach-test-twist beat structure, progression rules; authored to fit the CS level-editor data format.
    • Prerequisites: D1 ⭐ (mechanics and progression frame).
    • Feeds into: M2 #6 (the built level), CS M2 G2 ⭐ (editor content requirements).
    • Quality ladder: Base = coherent maps + curve. Good = levels teach mechanics without text. Exceptional = paper-playtested with documented iterations.
  7. Game Economy & Systems Balance Model — every resource has a source, a sink and a reason to exist.
    Details & dependencies
    • Deliverable: System diagram of all resources/mechanics and their interlocks; spreadsheet balance model with tunable variables; "no purposeless system" audit of the GDD.
    • Prerequisites: D1 ⭐ (systems defined).
    • Feeds into: M2 #7 (tuning), CS M1 G2 ⭐ (which values are data-driven).
    • Quality ladder: Base = model covers all systems. Good = interlocks explicit; model answers "what if" questions. Exceptional = model predictions checked against prototype play data.
  8. Narrative, Theme & World Concept — who am I, where am I, what must I do — answered with craft.
    Details & dependencies
    • Deliverable: Theme and tone definition, world/character bible (concise), narrative beats for the vertical slice, plan for in-game delivery (ambient, dialogue, cutscene).
    • Prerequisites: D1 ⭐ (theme/genre frame).
    • Feeds into: M2 #9 (cutscene presentation), Art track's art direction (or M1 #11 for no-Art teams).
    • Quality ladder: Base = coherent theme + beats. Good = narrative reinforces mechanics (ludonarrative fit). Exceptional = distinctive voice/world that strengthens the USP.
  9. Playtest Plan & First Playtests — structured observation, not "my friends liked it".
    Details & dependencies
    • Deliverable: Playtest protocol (goals, tasks, observation sheet, consent), at least 3 sessions run on the M1 prototype, findings log with severity ratings and design responses.
    • Prerequisites: M1 #2 (something playable) or D1 ⭐'s core-loop sketch.
    • Feeds into: M2 #5 (the full program), M2 D1 ⭐ (validation evidence).
    • Quality ladder: Base = protocol + 3 documented sessions. Good = findings triaged and visibly acted on. Exceptional = quantitative + qualitative methods combined.
  10. Accessibility & Readability Foundations — legible fonts, sane contrast, touch targets that fit thumbs.
    Details & dependencies
    • Deliverable: Accessibility guideline set for the project (type sizes, contrast ratios, colorblind-safe palette rules, touch-target minimums, input alternatives) applied to the M1 wireframes.
    • Prerequisites: M1 #4 (wireframes to apply it to).
    • Feeds into: M2 D1 ⭐ (polished UI must comply), M2 #10 (mobile ergonomics).
    • Quality ladder: Base = guidelines exist and wireframes comply. Good = verified with tools (contrast checkers, device tests). Exceptional = tested with users having relevant needs.
  11. Art Proof of Concept (conditional — mandatory pick for teams with NO Art student) — the design-owned substitute for the Art track's M1.
    Details & dependencies
    • Deliverable: Max-5-page Art POC (PDF): art references/concept art for characters or key objects, environments, menus/HUD; an art style guide covering ALL game elements; a UI style guide (fonts, themes, palettes, mood board). AI use for ideation allowed but every AI image annotated with the tool; AI assets may NOT ship in the product.
    • Prerequisites: D1 ⭐ (theme/genre), M1 #8 if picked.
    • Feeds into: M2 #11 (implementation quality), CS art pipeline goals.
    • Quality ladder: Base = complete POC with consistent references. Good = style guides give clear direction for all elements. Exceptional = original team-made mockups near industry quality.

Milestone 2 (Iterated Prototype phase)

  1. ⭐ UX Validation & Final Polished UI — tested with real users, polished to the "A" standard, running in the team's engine on the mobile target.
    Details & dependencies
    • Deliverable: Iterative UX testing across M2 with documented changes; complete menu functionality in the showcase build (start, pause, resume — exact labels, quit with confirmation, how-to-play); final polished UI implemented with the CS team — modern, animated, alive (not static screens); UI complies with the M1 accessibility guidelines; runs on the mobile target.
    • Prerequisites: M1 D1 ⭐, M1 #4 (flows/wireframes); CS M2 #5 or the CS GUI capability.
    • Feeds into: The trimester showcase; sets the visual/UX standard the T2 product must match.
    • Quality ladder: Base = all menus function correctly with standard labels/flows. Good = coherent, branded, animated UI; test-driven iterations documented. Exceptional = "2025-style" motion UI (Persona 5 reference) fully in-engine.
  2. Visual Branding & Style Guide — logo, typography, palette: one identity across game, UI and store presence.
    Details & dependencies
    • Deliverable: Logo/wordmark, typography system (max 3 fonts: headings, in-game, small text — policy-compliant), color palette, applied consistently across menus/HUD/marketing shots; style guide document.
    • Prerequisites: M1 D1 ⭐ (theme); M1 #11 or Art track (art direction).
    • Feeds into: M2 D1 ⭐ (polished UI uses it), the showcase presentation.
    • Quality ladder: Base = complete, consistent guide. Good = identity distinctive and applied everywhere. Exceptional = professional-grade brand system.
  3. Menu & HUD Motion Design — menus that build, respond and breathe; HUD states that animate meaningfully.
    Details & dependencies
    • Deliverable: Motion spec (build-on animations, hover/press states, transitions, HUD state changes like health-bar reactions) plus implementation with the CS team in the engine.
    • Prerequisites: M1 #4 (flows), M2 #2 (visual language); CS GUI/graphics goals.
    • Feeds into: M2 D1 ⭐ (its Exceptional rung lives here).
    • Quality ladder: Base = specified states for all interactive elements. Good = implemented animated feedback on menus and HUD. Exceptional = cohesive motion identity across the whole product.
  4. Onboarding & Tutorial Design — the first five minutes: teach without a wall of text.
    Details & dependencies
    • Deliverable: First-time-user experience design: contextual teaching moments, how-to-play screen content, safe failure spaces; validated with players who never saw the game.
    • Prerequisites: M1 #6 (teaching beats) or D1 ⭐; a playable M2 build.
    • Feeds into: M2 D1 ⭐ (how-to-play requirement), showcase demo readiness.
    • Quality ladder: Base = complete FTUE plan + how-to-play. Good = new players succeed unaided in tests. Exceptional = tutorial invisible — taught entirely through play.
  5. Playtesting & Iteration Program — the M2 engine of quality: test, triage, change, re-test.
    Details & dependencies
    • Deliverable: At least 1 structured session in M2 with persona-matched testers, metrics (task success, time, subjective ratings), a triaged findings backlog, and documented before/after changes.
    • Prerequisites: M1 #9 (protocol) or equivalent; M2 builds.
    • Feeds into: M2 D1 ⭐ (validation evidence), M2 #7 (tuning data).
    • Quality ladder: Base = 1 session, findings logged. Good = changes traceable to findings. Exceptional = multiple sessions with measurable improvement across iterations.
  6. Complete Basic Level & Vertical Slice Design — one level, whole experience: win, lose, retry.
    Details & dependencies
    • Deliverable: A complete basic level authored in the team's editor, with success and failure conditions and easy retry/continue; the vertical-slice experience arc (or an approved waiver for endless/no-fail designs).
    • Prerequisites: M1 #6 (level design) or D1 ⭐; CS M2 G2 ⭐ (editor).
    • Feeds into: The showcase demo; T2 full-level production.
    • Quality ladder: Base = playable complete level with win/lose. Good = paced, teaching, polished. Exceptional = a true vertical slice — all mechanics represented with intended experience quality.
  7. Systems Balance & Tuning — Data-Driven — turning the M1 model into a tuned game via the team's own data files.
    Details & dependencies
    • Deliverable: Tuning passes on the real build using the CS data-driven architecture (no code changes), tuning journal (what/why/result), final balance state for the showcase.
    • Prerequisites: M1 #7 (model); CS M1 G2 ⭐ (data files to tune).
    • Feeds into: Showcase play quality; M2 #5 feeds it data.
    • Quality ladder: Base = documented tuning to a stable state. Good = playtest-informed decisions. Exceptional = telemetry-assisted tuning.
  8. Game Feel & Feedback Pass ("Juice") — every action answers the player: visually, audibly, immediately.
    Details & dependencies
    • Deliverable: Feedback spec + implementation with CS: hit reactions, particles, screen effects, audio cues on meaningful interactions (objects, NPCs, items) serving the core mechanics.
    • Prerequisites: M1 #5 (feel spec); CS graphics/animation goals.
    • Feeds into: M2 D1 ⭐ and the showcase's perceived quality.
    • Quality ladder: Base = feedback on all core interactions. Good = cues are meaningful and legible, not noise. Exceptional = the game feels professional — feedback is a signature.
  9. Narrative Presentation & Cutscenes — context delivered with motion: scrolling, parallax, zoom — never static walls of text.
    Details & dependencies
    • Deliverable: Primitive narrative cutscenes or ambient storytelling answering "who am I / where am I / what must I do", using dynamic presentation; integrated in the build.
    • Prerequisites: M1 #8 (beats); art assets from Art track or M1 #11.
    • Feeds into: Showcase storytelling; T2 full narrative.
    • Quality ladder: Base = context delivered in-game. Good = dynamic presentation (parallax/zoom/scroll). Exceptional = presentation quality that elevates the whole product.
  10. Mobile UX Adaptation & Ergonomics — designed for thumbs, sessions and interruptions, on the reference device.
    Details & dependencies
    • Deliverable: Mobile ergonomics pass: thumb-zone layout, safe areas, session-length fit, interruption handling UX (calls/backgrounding), device testing on Pixel 8a/9a and/or mobile web.
    • Prerequisites: M1 #5, M1 #10; CS M2 G1 ⭐ (mobile build).
    • Feeds into: M2 D1 ⭐ (the polished UI must survive real hands on the real device).
    • Quality ladder: Base = layouts adapted and device-tested. Good = interruption/resume UX handled gracefully. Exceptional = one-handed play validated with users.
  11. Art Implementation Quality (conditional — mandatory pick for teams with NO Art student) — the design-owned substitute for the Art track's M2.
    Details & dependencies
    • Deliverable: All implemented art aligned with the Art POC (no placeholders): readable environments, critical sprites/animations for key actions, consistent scaling (matching pixel sizes, no distortion), UI states implemented; all shipped art team-made.
    • Prerequisites: M1 #11 (Art POC); CS art pipeline goals.
    • Feeds into: Showcase visual quality; T2 art production plan.
    • Quality ladder: Base = critical art implemented, readable, consistent. Good = environment alive (layering, ambient animation). Exceptional = near-industry quality throughout.

Category adaptations

Pattern approved by the Design instructor (Education, July 2026); the subsections follow the approved model. The pool above is canonical and Games-flavored; most picks are universal design disciplines and apply to every category as-is (user research #3, UX flows #4, playtests #9, accessibility #10; branding M2 #2, motion M2 #3, onboarding M2 #4, playtesting program M2 #5, mobile ergonomics M2 #10, and both conditionals). The subsections below give only the category readings for D1 ⭐ and the picks whose meaning shifts. Unlisted picks apply unchanged.

Education + Training

  • D1 ⭐ (M1) reads as a Learning Product Design Document: learning objectives and target learners replace genre/audience; the problem statement is the skill gap the product closes; feasibility unchanged. D1 ⭐ (M2): UX validation with real learners where possible; final polished UI unchanged.
  • M1 #2 Core-Loop Interactive Prototype → prototype of the core exercise loop (one drill/exercise playable end to end: attempt → feedback → next).
  • M1 #5 Control Mapping & Game Feel Specification → Interaction & Feedback Specification: input mapping for drills (touch/keyboard), feedback timing rules (correct/incorrect within 100 ms), error-tolerance conventions.
  • M1 #6 Level & Progression Design → Curriculum & Difficulty Progression: lesson sequencing, difficulty curves, mastery gates, remediation paths.
  • M1 #7 Game Economy & Systems Balance Model → Scoring, Streaks & Motivation Systems: points, streaks, spaced-repetition parameters, reward cadence — tuned against the learning goals, not against engagement alone.
  • M1 #8 Narrative, Theme & World Concept → Content Theme & Scenario Design: story wrappers for lessons, scenario design (e.g. driving-theory situations), age-appropriate tone.
  • M2 #6 Complete Basic Level & Vertical Slice → one complete lesson unit polished end to end (intro → exercises → assessment → results).
  • M2 #7 Systems Balance & Tuning → difficulty-curve tuning from playtest/telemetry data (error rates per exercise drive re-sequencing).
  • M2 #8 Game Feel & Feedback Pass → feedback & celebration pass: correct/incorrect/progress moments feel rewarding without distracting from learning.
  • M2 #9 Narrative Presentation & Cutscenes → guided sequences & worked examples: intro sequences, demonstration replays, tutor walk-throughs.

Simulator

  • D1 ⭐ (M1) reads as a Simulation Product Design Document: the modeled system, who consumes the insights and for what decisions, fidelity-vs-usability trade-offs, feasibility. D1 ⭐ (M2): UX validation with users interpreting results; final polished UI unchanged.
  • M1 #2 Prototype → the configure → run → read-results loop, interactive end to end.
  • M1 #5 Control Mapping & Game Feel → Run-Control & Visualization Spec: play/pause/speed/step controls, camera & inspection interactions, live-view readability rules.
  • M1 #6 Level & Progression → Scenario Set Design: authored scenarios from baseline to stress and edge cases, ordered for demos.
  • M1 #7 Economy & Balance → Parameter Model Design: parameter ranges, defaults and plausibility bounds, designed with the CS team.
  • M1 #8 Narrative/Theme → Case-Study Framing: the real-world story each scenario tells (a city, an ecosystem), naming and visual metaphor.
  • M2 #6 Vertical Slice → one polished experiment end to end (configure → run → batch → report).
  • M2 #7 Balance & Tuning → parameter tuning against reference data or documented expectations.
  • M2 #8 Feedback Pass → legibility pass on the live view: transitions, highlights, alerts that make the simulation satisfying to watch.
  • M2 #9 Cutscenes → guided tours / annotated replays of notable runs.

Fintech

  • D1 ⭐ (M1) reads as a Financial Product Design Document: user goals (learn / monitor / decide), trust and clarity requirements, virtual-money disclaimers, feasibility. D1 ⭐ (M2) unchanged.
  • M1 #2 Prototype → the core financial loop (observe → decide → act → review outcome).
  • M1 #5 Control Mapping & Game Feel → Interaction & Data-Density Spec: chart interactions, order/entry ergonomics, error prevention for money-like actions.
  • M1 #6 Level & Progression → User Journey & Scenario Design: onboarding to first action; scripted market scenarios (calm, volatile, crash).
  • M1 #7 Economy & Balance → Economy & Risk Model Design: virtual portfolio rules, fees, limits — with the CS simulation.
  • M1 #8 Narrative/Theme → Product Voice & Scenario Theming: sober vs playful tone, market narratives for demos.
  • M2 #6 Vertical Slice → one complete workflow polished (e.g. fund → trade → report).
  • M2 #7 Balance & Tuning → tuning fees/volatility/difficulty from playtests.
  • M2 #8 Feedback Pass → clarity pass: confirmations, alerts, P&L moments — no dark patterns.
  • M2 #9 Cutscenes → guided walkthroughs / annotated market replays.

GovTech

  • D1 ⭐ (M1) reads as a Service Design Document: the service blueprint across the citizen/customer side and the staff/ops side, stakeholders, accessibility mandate, feasibility. D1 ⭐ (M2) unchanged.
  • M1 #2 Prototype → the service flow (submit → track → resolve), both roles clickable.
  • M1 #5 Control Mapping & Game Feel → Form & Status Ergonomics Spec: form patterns, validation behaviour, status feedback conventions.
  • M1 #6 Level & Progression → Service Journey Design: end-to-end citizen journeys incl. rejection/appeal paths; staff workload paths.
  • M1 #7 Economy & Balance → Rules & Priority Model Design: queue priorities, SLA thresholds, escalation rules — with the CS engine.
  • M1 #8 Narrative/Theme → Civic Identity & Plain Language: trust cues, multilingual tone, content standards.
  • M2 #6 Vertical Slice → one service polished end to end (citizen submits → staff resolves → citizen notified).
  • M2 #7 Balance & Tuning → threshold/priority tuning under simulated load.
  • M2 #8 Feedback Pass → status & notification pass: clarity and reassurance at every state change.
  • M2 #9 Cutscenes → role onboarding walkthroughs and the showcase demo storyline.

Social Media

  • D1 ⭐ (M1) reads as a Community Product Design Document: the target community, interaction norms, safety-by-design, feasibility. D1 ⭐ (M2) unchanged.
  • M1 #2 Prototype → the core social loop (post → react → notify → return).
  • M1 #5 Control Mapping & Game Feel → Feed & Composer Ergonomics Spec: gestures, composer flow, notification behaviour.
  • M1 #6 Level & Progression → Community Journey Design: new-user journey, empty states, seeding strategy for believable demos.
  • M1 #7 Economy & Balance → Engagement & Reputation Model: points/karma/limits balanced against abuse — with the moderation pipeline.
  • M1 #8 Narrative/Theme → Community Identity & Tone: naming, voice, presentation of guidelines.
  • M2 #6 Vertical Slice → one community flow polished (e.g. create event → invite → attend → recap).
  • M2 #7 Balance & Tuning → ranking/limit tuning from usage tests.
  • M2 #8 Feedback Pass → reactions, streaks and presence moments that reward participation.
  • M2 #9 Cutscenes → onboarding stories over seeded community content.

CMS

  • D1 ⭐ (M1) reads as an Authoring Product Design Document: author and audience personas, content-model concepts, WYSIWYG expectations, feasibility. D1 ⭐ (M2) unchanged.
  • M1 #2 Prototype → the authoring loop (create → edit → preview → publish).
  • M1 #5 Control Mapping & Game Feel → Editor Interaction Spec: selection, drag-drop, inline editing conventions, undo expectations.
  • M1 #6 Level & Progression → Template & Content Structure Design: page archetypes, block taxonomy, starter templates.
  • M1 #7 Economy & Balance → Publishing Workflow Design: draft/review/publish states, roles, versioning UX.
  • M1 #8 Narrative/Theme → Theme & Demo Content Strategy: the demo site's content story, theme direction.
  • M2 #6 Vertical Slice → one authoring-to-published-site journey polished with real content.
  • M2 #7 Balance & Tuning → workflow tuning from author tests (friction points, defaults).
  • M2 #8 Feedback Pass → editor feel pass: micro-feedback, drag ghosting, save states.
  • M2 #9 Cutscenes → guided editor tours and template showcase sequences.

Immersive Experience

Closest to Games — most picks apply nearly as written; only these readings shift:

  • D1 ⭐ (M1) reads as an Experience Design Document: theme, emotional arc, context/space of use, feasibility. D1 ⭐ (M2) unchanged.
  • M1 #2 Prototype → one interactive scene/moment working end to end.
  • M1 #6 Level & Progression → Experience Flow & Pacing: scene sequence, dwell-time design, transitions.
  • M1 #7 Economy & Balance → Reactivity Model: mapping inputs (audio, motion, touch) to response intensities.
  • M2 #6 Vertical Slice → one complete scene/chapter polished.
  • M2 #7 Balance & Tuning → reactivity tuning (thresholds, easing) from observing visitors.

Creative Tools & Media Production

  • D1 ⭐ (M1) reads as a Tool Design Document: target creators, core workflows, competitive tool analysis, feasibility. D1 ⭐ (M2): UX validation with real creators; final polished UI unchanged.
  • M1 #2 Prototype → the core creation workflow (create → edit → export happy path).
  • M1 #5 Control Mapping & Game Feel → Tool Interaction Spec: shortcuts, modes, direct-manipulation conventions, touch adaptation.
  • M1 #6 Level & Progression → Workflow & Learning-Curve Design: novice→power-user progression, defaults, preset ladder.
  • M1 #7 Economy & Balance → Preset & Constraint Model: sensible parameter ranges, starter content balance.
  • M1 #8 Narrative/Theme → Product Identity & Showcase Content Plan: the tool's voice and what gets made with it for the demo.
  • M2 #6 Vertical Slice → one real artifact produced end to end in the tool, polished.
  • M2 #7 Balance & Tuning → ergonomics tuning from creator tests (click counts, friction).
  • M2 #8 Feedback Pass → tool feel pass: snapping, ghosting, micro-animations, satisfying manipulation.
  • M2 #9 Cutscenes → tutorial sequences and made-with-it showcase reels.
